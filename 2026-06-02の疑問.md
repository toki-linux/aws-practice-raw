
# RDSへの接続で、なぜS3 Gateway Endpointが必要になったのか

## 概要

Private Subnet内のEC2から、RDS for MySQLへ接続する実践を行った。

構築途中で `S3 Gateway Endpoint` を追加したため、次の疑問が生まれた。

> RDSへ接続する実践なのに、なぜS3が必要なのか？

結論から言うと、**RDSへの接続自体にS3 Gateway Endpointは必要ない**。

S3 Gateway Endpointが必要になった理由は、Private EC2へMySQLクライアントをインストールする際に、インストール用ファイルをS3から取得する必要があったためである。

---

## RDSへの接続経路

EC2からRDSへ接続するときの経路は、以下のとおり。

```text
Private EC2
↓ TCP 3306
RDS Endpoint
↓
RDS for MySQL
```

RDSはVPC内に配置されているため、EC2とRDSの間で直接通信できる。

この通信では、S3 Gateway Endpointを通らない。

必要なのは、RDS用Security Groupの許可設定である。

```text
Inbound Rule
Type：MySQL/Aurora
Protocol：TCP
Port：3306
Source：private-sg
```

`private-sg` は、RDSへ接続するEC2に付与したSecurity Groupである。

---

## なぜS3が登場したのか

Private EC2からRDSを操作するためには、EC2側で `mysql` コマンドを使えるようにする必要があった。

そのため、Amazon Linux 2023のEC2内で以下を実行した。

```bash
sudo dnf install mariadb105
```

このコマンドは、EC2へMariaDBクライアントをインストールするためのもの。

MariaDBクライアントを導入すると、以下のように `mysql` コマンドを使ってRDSへ接続できる。

```bash
mysql -h <RDSエンドポイント> -P 3306 -u admin -p
```

---

## インストール用ファイルはどこにあるのか

`dnf install mariadb105` を実行すると、EC2はAmazon Linuxのパッケージリポジトリから、インストールに必要なファイルを取得する。

今回使用したAmazon Linux 2023では、パッケージ取得時にAWSが管理するS3バケットへアクセスする必要がある。

イメージは以下のとおり。

```text
Amazon Linuxの運営側
↓
インストール用ファイルをS3へ配置

Private EC2
↓
mariadb105をインストールしたい
↓
S3上のファイルを取得
```

S3は、RDS専用のサービスではない。

S3はAWS上の汎用的なオブジェクトストレージであり、AWS自身もソフトウェア配布用の保管場所として利用している。

---

## なぜ通常はS3を意識しないのか

インターネットへ接続できるEC2であれば、パッケージの保存場所を意識せずにインストールできる。

```text
Public EC2
または
NAT Gatewayを利用できるPrivate EC2
↓
パッケージリポジトリへ到達
↓
dnf install 成功
```

しかし、今回のPrivate EC2には、インターネットへ出るための経路を用意していなかった。

```text
Private EC2
├── Public IPなし
├── NAT Gatewayなし
└── Internet Gatewayへ向かう経路なし
```

そのため、Private EC2からS3へ接続するための専用経路を追加する必要があった。

それが、`S3 Gateway Endpoint` である。

```text
Private EC2
↓
S3 Gateway Endpoint
↓
AWS管理のS3バケット
↓
mariadb105のファイルを取得
```

---

## S3 Gateway Endpointの役割

S3 Gateway Endpointは、VPC内のリソースからS3へプライベートに接続するための経路である。

Internet GatewayやNAT Gatewayを使わずに、S3へアクセスできる。

作成時には、Private EC2が利用するRoute Tableへ関連付ける。

イメージは以下のとおり。

```text
Private SubnetのRoute Table
↓
S3向けの通信
↓
S3 Gateway Endpoint
↓
S3
```

---

## RDS Endpointとの違い

今回登場した「Endpoint」は、それぞれ役割が異なる。

| 種類 | 役割 | 今回の用途 |
|---|---|---|
| RDS Endpoint | 作成したデータベースへ接続するための接続先 | `mysql` コマンドでRDS for MySQLへ接続 |
| S3 Gateway Endpoint | VPC内からS3へプライベートに接続するための経路 | `mariadb105` のインストール用ファイルを取得 |
| RDS用Interface Endpoint | RDS APIへプライベートに接続するための入口 | 今回は未使用 |

---

## RDS用Interface Endpointでは代用できない理由

RDS用Interface Endpointは、RDSの管理APIへプライベートに接続するためのもの。

たとえば、AWS CLIやSDKから以下のような操作を行う際に利用する。

```text
RDSインスタンスの作成
RDSインスタンスの変更
RDSインスタンスの削除
RDSインスタンスの状態確認
```

一方で、RDS for MySQLのデータベースへ接続するときは、RDS EndpointへTCP 3306で接続する。

また、RDS用Interface Endpointには、MariaDBクライアントのインストール用ファイルを配布する機能はない。

そのため、S3 Gateway Endpointの代わりにはならない。

---

## 今回の流れ

今回の作業を順番に整理すると、以下のようになる。

```text
1. RDS for MySQLを作成した
↓
2. Private EC2からRDSを操作したい
↓
3. EC2にmysqlコマンドが入っていなかった
↓
4. mariadb105をインストールしようとした
↓
5. インストール用ファイルの取得先としてS3への接続が必要になった
↓
6. Private EC2にはインターネットへ出る経路がなかった
↓
7. S3 Gateway Endpointを追加した
↓
8. mariadb105を取得できた
↓
9. mysqlコマンドでRDSへ接続した
```

---

## 誤解しやすいポイント

### 誤った理解

```text
RDSを使うためには、S3 Gateway Endpointが必要
```

### 正しい理解

```text
RDSへの接続自体に、S3 Gateway Endpointは不要

ただし今回は、
EC2へMySQLクライアントをインストールする際に、
S3上のファイルを取得する必要があった

そのため、
Private EC2からS3へ接続する経路として
S3 Gateway Endpointを追加した
```

---

## 一言でまとめると

```text
S3
→ パッケージリポジトリの実体となるファイルの保管場所

S3 Gateway Endpoint
→ Private EC2が、その保管場所へ接続するための専用経路

RDS Endpoint
→ EC2が、作成したMySQLデータベースへ接続するための住所
```

RDSを利用するからS3が必要なのではない。

**EC2へMySQLクライアントを追加するために、S3上のインストール用ファイルを取得する必要があった。**
