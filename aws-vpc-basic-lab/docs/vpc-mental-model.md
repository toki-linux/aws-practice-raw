# VPC Mental Model

## 概要

このドキュメントは、VPC・サブネット・Internet Gateway・ルートテーブル・セキュリティグループの関係を、自分なりに整理した学習メモです。

実際に自作VPCを作成し、EC2を起動してインターネット疎通を確認したことで、各リソースの役割が少しずつ整理できました。

---

## VPCとは何か

VPCは、AWS上に作成する自分専用のネットワーク空間です。

EC2インスタンスは、単体で空中に存在しているわけではなく、必ずどこかのVPCの中に配置されます。

```text
VPC
└── Subnet
    └── EC2
```

以前は、EC2を作成するとVPCが作られるようなイメージを持っていました。

しかし、実際の階層としてはVPCの方が上です。

```text
VPCが先
EC2はその中に置かれる
```

---

## なぜVPCが必要なのか

VPCは、AWS上のリソースをネットワーク的に分けるために存在します。

もしVPCのような区切りがなければ、すべてのEC2やAWSリソースが同じネットワーク上に雑に置かれてしまいます。

VPCがあることで、以下を自分で管理できます。

- IPアドレスの範囲
- サブネットの分割
- インターネットに出すリソース
- 外部から見せないリソース
- 通信経路
- セキュリティ境界

つまりVPCは、AWS上のネットワークの土台です。

---

## VPCとEC2の関係

EC2インスタンスは、VPCの中のサブネットに配置されます。

```text
VPC
└── Subnet
    └── EC2 Instance
```

EC2を起動するときは、どのVPCのどのサブネットに置くかを決める必要があります。

デフォルトVPCを使う場合は、AWSが最初から用意してくれているため意識しにくいです。

しかし、自作VPCを作ると、EC2がVPC内に配置されるリソースであることがわかりやすくなります。

---

## サブネットとは何か

サブネットは、VPCのIPアドレス範囲をさらに小さく区切ったものです。

今回の例では、VPC全体に以下のCIDRを設定しました。

```text
10.0.0.0/16
```

その中に、以下のサブネットを作成しました。

```text
10.0.1.0/24
```

イメージは以下です。

```text
VPC: 10.0.0.0/16
└── Subnet: 10.0.1.0/24
```

VPCが大きな箱で、サブネットはその中の小部屋です。

---

## パブリックサブネットとは何か

パブリックサブネットは、インターネットと通信できる経路を持つサブネットです。

ただし、サブネット名に `public` と付けただけではパブリックサブネットにはなりません。

パブリックサブネットになるには、主に以下が必要です。

```text
1. VPCにInternet Gatewayがアタッチされている
2. サブネットに関連付いたルートテーブルに 0.0.0.0/0 → Internet Gateway がある
3. EC2にパブリックIPがある
4. セキュリティグループで必要な通信が許可されている
```

つまり、パブリックサブネットかどうかは名前ではなく、通信経路で決まります。

---

## Internet Gatewayとは何か

Internet Gatewayは、VPCがインターネットと通信するための出口です。

```text
VPC
└── Internet Gateway
    └── Internet
```

ただし、Internet Gatewayを作成してVPCにアタッチしただけでは、EC2が自動的にインターネットへ通信できるわけではありません。

サブネットからInternet Gatewayへ向かうルートが必要です。

---

## ルートテーブルとは何か

ルートテーブルは、通信をどこへ流すかを決める道案内です。

今回のルートテーブルは以下です。

```text
10.0.0.0/16 → local
0.0.0.0/0  → Internet Gateway
```

それぞれの意味は以下です。

```text
10.0.0.0/16 → local
VPC内の通信はVPC内で処理する

0.0.0.0/0 → Internet Gateway
それ以外の外向き通信はInternet Gatewayへ流す
```

---

## 0.0.0.0/0 の意味

`0.0.0.0/0` は、すべてのIPv4アドレスを表します。

ルートテーブルで以下のように設定すると、

```text
0.0.0.0/0 → Internet Gateway
```

VPC内で行き先がローカルではない通信は、Internet Gatewayへ向かいます。

これにより、EC2からインターネットへ通信できるようになります。

---

## セキュリティグループとは何か

セキュリティグループは、EC2に対する通信の許可ルールです。

今回の例では、SSH接続のために以下を許可しました。

```text
Type: SSH
Protocol: TCP
Port: 22
Source: My IP
```

セキュリティグループは、EC2の入口を制御するイメージです。

```text
Security Group
└── EC2への通信を許可・拒否する
```

外部からSSH接続するには、以下がそろっている必要があります。

```text
EC2にパブリックIPがある
Security GroupでSSHが許可されている
サブネットがInternet Gatewayへのルートを持っている
```

---

## パブリックIPとは何か

パブリックIPは、インターネット側からEC2を見つけるための住所です。

EC2がプライベートIPだけを持っている場合、VPC内では通信できますが、インターネットから直接SSH接続することはできません。

```text
Private IP:
VPC内で使う住所

Public IP:
インターネット側から使う住所
```

今回のEC2では、パブリックIPを有効にしてSSH接続しました。

---

## 今回の通信経路

EC2からインターネットへ通信するときの流れは以下です。

```text
EC2
↓
Public Subnet
↓
Route Table
↓
Internet Gateway
↓
Internet
```

`curl -I https://aws.amazon.com` を実行し、HTTPステータスコード `200` が返ってきたことで、この経路が正しく機能していることを確認できました。

---

## default セキュリティグループが増えた理由

自作VPCを作成したあと、セキュリティグループ一覧に `default` が2つ表示されました。

これは、VPCごとに `default` セキュリティグループが自動作成されるためです。

```text
Default VPC
└── default Security Group

自作VPC
└── default Security Group
```

そのため、`default` が複数あること自体は正常です。

見分けるときは、セキュリティグループ名ではなく、所属している `VPC ID` を確認します。

---

## 削除時の考え方

VPCを削除するときは、VPC本体から先に削除するのではなく、中にあるリソースから削除します。

```text
EC2
↓
Security Group
↓
Subnet
↓
Route Table
↓
Internet Gateway
↓
VPC
```

考え方は以下です。

```text
使っているものを先に消す
使われていた入れ物を後で消す
```

VPCは土台なので、最後に削除します。

---

## 自分なりの理解

今回の実践を通して、以下のように理解しました。

```text
VPC = AWS上の自分専用ネットワークの箱
Subnet = VPC内を分けた小部屋
EC2 = サブネット内に置くサーバ
Internet Gateway = インターネットへの出口
Route Table = 通信の道案内
Security Group = EC2の入口許可
Public IP = 外からEC2を見つける住所
```

EC2がインターネットへ通信できるには、これらが正しくつながっている必要があります。

---

## まとめ

今回の実践で、VPCの基本構成を自分で作成し、EC2からインターネット疎通できることを確認しました。

特に重要だと感じたのは、以下です。

- VPCはEC2より上位のネットワーク土台
- EC2はVPC内のサブネットに配置される
- Internet Gatewayだけではインターネット通信はできない
- ルートテーブルでInternet Gatewayへの道を設定する必要がある
- パブリックサブネットは名前ではなくルートで決まる
- セキュリティグループはEC2への入口制御
- VPC削除時は中身から削除し、VPC本体は最後に消す

今回の学習によって、EC2を起動するときに裏側でどのようなネットワーク構成が必要なのかを理解しやすくなりました。
