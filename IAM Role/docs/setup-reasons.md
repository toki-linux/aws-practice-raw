# EC2 + IAM Role + S3 実践：設定理由まとめ

## 目的

このドキュメントでは、EC2からIAM Roleを使ってS3へアクセスする実践で、各設定をなぜそのようにしたのかを整理する。

---

## S3バケットを作成した理由

今回の目的は、EC2にIAM Roleを付けることで、EC2からAWSサービスへアクセスできることを確認することだった。

その確認先としてS3を使った。

理由は以下。

```text
S3はファイルを置いて確認しやすい
AWS CLIで操作しやすい
読み取り権限の確認がしやすい
EC2からAWSサービスを操作する練習に向いている

今回の実践では、S3そのものを深く学ぶことよりも、EC2にIAM Roleを付けるとAWSサービスを操作できることを確認することを重視した。

S3を非公開にした理由

S3バケットのパブリックアクセスはすべてブロックした。

理由は、今回の目的がWeb公開ではないため。

今回の目的：EC2からS3を読むこと
今回やらないこと：S3をインターネットに公開すること

S3を公開する必要がない場合は、パブリックアクセスをブロックしておく方が安全。

バケットタイプを「汎用」にした理由

S3バケットタイプは「汎用」を選んだ。

理由は、通常のファイル保存・読み取りの練習には汎用バケットで十分だから。

汎用バケット
→ 通常のS3利用向け

ディレクトリバケット
→ より特殊な低レイテンシー用途向け

今回はIAM Roleの練習が目的なので、特別なバケットタイプは不要だった。

バケット名前空間を「グローバル名前空間」にした理由

通常のS3バケットは、グローバルで一意な名前が必要になる。

そのため、他のAWS利用者とかぶらないバケット名にした。

例。

toki-iam-role-practice-日付

S3バケット名は世界中で一意にする必要があるため、名前や日付を入れて重複しにくくした。

test.txtを置いた理由

S3アクセス確認用に test.txt を置いた。

理由は、EC2から以下の流れを確認するため。

S3バケットが見える
S3バケット内のファイルが見える
S3からEC2へファイルをコピーできる
コピーしたファイルの中身を確認できる

単に aws s3 ls でバケット一覧が見えるだけではなく、実際にファイル取得まで確認することで、S3読み取り権限が使えていることを確認した。

IAM Roleを作成した理由

EC2にS3を読み取る権限を渡すために、IAM Roleを作成した。

IAM Roleを使う理由は、EC2内にアクセスキーを保存しなくて済むため。

悪い例。

EC2内にアクセスキーを書く
aws configureでアクセスキーを設定する

良い例。

EC2にIAM Roleを付ける
アクセスキーなしでAWS CLIを使う

IAM Roleを使うことで、認証情報をEC2内に直接保存せずにAWSサービスへアクセスできる。

信頼されたエンティティをEC2にした理由

IAM Role作成時に、信頼されたエンティティとしてEC2を選んだ。

理由は、今回そのRoleを使う主体がEC2だから。

誰がこのRoleを使うのか？
→ EC2

そのため、ユースケースは以下にした。

AWSのサービス
→ EC2

これは「このRoleはEC2インスタンスが使うRoleです」という設定。

AmazonS3ReadOnlyAccessを付けた理由

IAM Roleには AmazonS3ReadOnlyAccess を付与した。

理由は、今回必要なのがS3の読み取りだけだったから。

今回やりたいこと。

S3バケット一覧を見る
S3バケット内のファイルを見る
S3からファイルを取得する

今回やらないこと。

S3へファイルをアップロードする
S3のファイルを削除する
S3バケットを作成・削除する

そのため、書き込み権限や削除権限は不要。

必要最小限に近い形で、S3読み取り権限のみを付けた。

EC2作成時にIAM Roleを付けた理由

EC2作成時に、IAMインスタンスプロフィールとして作成したIAM Roleを設定した。

理由は、EC2からAWS CLIを実行したときに、そのRoleの権限でAWS APIを呼び出せるようにするため。

EC2
↓
IAM Roleを使用
↓
S3へアクセス

この設定により、EC2内で aws configure をしなくても、AWS CLIからS3へアクセスできるようになる。

セキュリティグループでSSHのみ許可した理由

今回のEC2では、セキュリティグループでSSHのみを許可した。

理由は、今回の目的がWebサーバ公開ではなく、S3アクセス確認だったから。

必要な通信：SSH接続
不要な通信：HTTPアクセス

そのため、HTTP 80番は開放しなかった。

SSH 22番：許可
HTTP 80番：不要なので許可しない

必要なポートだけ開けることで、余計な公開範囲を減らした。

nginxをインストールしなかった理由

今回の実践ではnginxをインストールしなかった。

理由は、今回の目的がWebページ表示ではなく、EC2からS3へIAM Roleでアクセスすることだったから。

前回までの実践ではnginxを使ってWebサーバ構築を行ったが、今回は以下が主目的。

IAM Role
AWS CLI
S3アクセス
アクセスキーなしの認証

目的に関係しない作業を増やさないため、nginxは入れなかった。

AWS CLIをインストールした理由

EC2からS3へアクセスするために、AWS CLIをインストールした。

AWS CLIを使うことで、EC2の中から以下の操作ができる。

自分がどのIAM Roleで認証されているか確認する
S3バケット一覧を表示する
S3バケット内のファイルを表示する
S3からファイルをコピーする

今回使った主なコマンド。

aws sts get-caller-identity
aws s3 ls
aws s3 ls s3://バケット名/
aws s3 cp s3://バケット名/test.txt .
aws configureをしなかった理由

今回、EC2内で aws configure は実行しなかった。

理由は、アクセスキーを使わずにIAM Roleで認証することが目的だったから。

aws configure を使うと、アクセスキーやシークレットアクセスキーをEC2内に設定する流れになる。

しかし、今回確認したいことは以下。

EC2にIAM Roleを付ければ、アクセスキーなしでAWS CLIが使える

そのため、あえて aws configure は使わなかった。

aws sts get-caller-identityを実行した理由

aws sts get-caller-identity を実行した理由は、EC2がどの権限でAWSにアクセスしているか確認するため。

期待する結果は、作成したIAM Roleが表示されること。

例。

assumed-role/EC2-S3-ReadOnly-Role/...

これにより、EC2がIAM Roleを使ってAWSにアクセスしていることを確認できる。

aws s3 lsを実行した理由

aws s3 ls を実行した理由は、EC2からS3バケット一覧を取得できるか確認するため。

aws s3 ls

このコマンドで作成したS3バケットが表示されれば、EC2からS3への読み取りアクセスができていると分かる。

aws s3 ls s3://バケット名/ を実行した理由

バケット一覧だけでなく、バケット内のファイルも確認した。

aws s3 ls s3://バケット名/

このコマンドにより、S3バケット内にアップロードした test.txt が見えるか確認した。

aws s3 cpを実行した理由

S3上のファイルをEC2へコピーするために、以下を実行した。

aws s3 cp s3://バケット名/test.txt .

このコマンドにより、S3上の test.txt をEC2のカレントディレクトリへ取得した。

ここで重要なのは、ls と cp の違い。

aws s3 ls
→ S3上のファイル一覧を見るだけ

aws s3 cp
→ S3からEC2へファイルをコピーする

ls だけではEC2上にファイルは作成されない。

cat test.txtを実行した理由

S3からEC2へコピーしたファイルの中身を確認するために、以下を実行した。

cat test.txt

これにより、S3から取得したファイルがEC2上に存在し、中身も正しく読めることを確認した。

実践後にリソースを削除した理由

課金を抑えるため、実践後に作成したリソースを削除した。

削除したもの。

EC2インスタンス
S3内のtest.txt
S3バケット
IAM Role

EC2は起動時間に応じて料金が発生するため、実践後に終了した。

S3バケットも練習用だったため、中身を削除してからバケットを削除した。

IAM Role自体は基本的に課金対象ではないが、練習用に作成したため削除した。

今回の設定全体の考え方

今回の設定は、以下の考え方で行った。

必要なものだけ作る
不要な通信は開けない
アクセスキーはEC2内に置かない
権限は読み取りだけにする
使い終わったリソースは削除する

今回の一番重要なポイントは以下。

EC2にIAM Roleを付けると、アクセスキーを保存しなくてもAWS CLIでS3を操作できる
