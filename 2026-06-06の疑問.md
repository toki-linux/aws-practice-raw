
---

# メトリクスとは？

## 概要

メトリクスは、システムの状態を数値で表した監視データ。

```text
メトリクス
=
監視に使う数値データ
```

---

## 例

EC2では、たとえば以下のような数値を確認できる。

```text
CPU使用率
ネットワーク受信量
ネットワーク送信量
ディスクの読み取り回数
ディスクの書き込み回数
```

数値の変化を確認することで、EC2に問題が起きていないか判断しやすくなる。

---

## CloudWatchとの関係

Amazon CloudWatchは、AWSリソースやアプリケーションの状態を監視するサービス。

```text
AWSリソース
↓
メトリクスを収集
↓
CloudWatch
↓
グラフ・アラームなどで確認
```

---

# 標準メトリクスとは？

## 初心者向けの理解

標準メトリクスは、AWSサービスが最初から提供している監視データ。

```text
標準メトリクス
=
AWSが最初から収集・提供している監視データ
```

たとえば、EC2を作成すると、CloudWatchでCPU使用率などを確認できる。

自分で監視プログラムを作らなくても、基本的な状態を確認できる。

---

## EC2で確認できる代表例

```text
CPUUtilization
→ CPU使用率

NetworkIn
→ EC2が受信したネットワーク通信量

NetworkOut
→ EC2が送信したネットワーク通信量

DiskReadOps
→ ディスクの読み取り回数

DiskWriteOps
→ ディスクの書き込み回数

StatusCheckFailed
→ EC2のステータスチェック失敗
```

---

## 注意点：メモリ使用率は標準では取得できない

EC2のメモリ使用率は、標準メトリクスに含まれない。

理由は、AWS側からEC2のOS内部にある細かな情報までは見えないため。

```text
AWS側から見えるもの
→ CPU使用率
→ ネットワーク通信量
→ EC2の状態

OS内部で確認するもの
→ メモリ使用率
→ ディスク使用率
→ プロセスの状態
```

メモリ使用率などをCloudWatchへ送るには、CloudWatch Agentなどを利用する。

```text
EC2
└── CloudWatch Agent
    ↓
メモリ使用率などを収集
    ↓
CloudWatchへ送信
```

---

# 詳細モニタリングとは？

## 概要

EC2では、標準で基本モニタリングが有効になっている。

基本モニタリングでは、メトリクスが通常5分間隔で送られる。

詳細モニタリングを有効にすると、1分間隔で確認できる。

```text
基本モニタリング
→ 通常5分間隔

詳細モニタリング
→ 1分間隔
```

---

## 注意点

詳細モニタリングを有効にしても、メモリ使用率が自動で追加されるわけではない。

```text
詳細モニタリング
→ 既存のEC2メトリクスを細かい間隔で取得する

CloudWatch Agent
→ OS内部の追加情報を取得する
```

---

# カスタムメトリクスとは？

## 概要

カスタムメトリクスは、自分でCloudWatchへ送信する監視データ。

```text
カスタムメトリクス
=
利用者が独自にCloudWatchへ送る監視データ
```

---

## 例

```text
メモリ使用率
ディスク使用率
ログイン回数
注文数
アプリケーション独自のエラー件数
```

---

## 標準メトリクスとの違い

| 種類 | 誰が用意するか | 例 |
|---|---|---|
| 標準メトリクス | AWSサービス側 | CPU使用率、ネットワーク通信量 |
| カスタムメトリクス | 利用者側 | メモリ使用率、注文数、独自エラー数 |

---

# 標準解像度と高解像度

CloudWatchには、メトリクスの保存間隔に関する別の用語もある。

```text
標準解像度
高解像度
```

これは、「AWSが自動収集するか」という話とは別。

```text
AWSが標準で提供するメトリクスか
→ 誰が収集・提供するかの話

標準解像度か、高解像度か
→ どの細かさで保存するかの話
```

---

## 標準解像度

```text
標準解像度
→ 通常は1分単位で保存する
```

---

## 高解像度

```text
高解像度
→ 最短1秒単位で保存できる
```

短い時間に大きく変化する値を詳しく確認したい場合に使う。

# nginxのドキュメントルートとは？

## 概要

ドキュメントルートは、Webサーバが公開するHTMLファイルなどを置く場所。

```text
ドキュメントルート
=
ブラウザへ返すWebページのファイルを置く場所
```

---

# `/var/www/html` が存在しなかった理由

## 状況

Ubuntuでnginxを使ったときは、以下のディレクトリへHTMLファイルを置いていた。

```text
/var/www/html
```

しかし、Amazon Linux 2023でnginxをインストールしたあとに確認すると、以下が存在しなかった。

```text
/var/www/
/var/www/html
```

---

## 結論

OSやディストリビューション、パッケージの初期設定によって、nginxのドキュメントルートが異なるため。

Amazon Linux系でインストールしたnginxでは、以下が使われることがある。

```text
/usr/share/nginx/html
```

---

## OSによる違い

```text
Ubuntu系でよく見る配置
└── /var/www/html

Amazon Linux系でよく見る配置
└── /usr/share/nginx/html
```

どちらも間違いではない。

---

## 重要なポイント

nginxは、必ず `/var/www/html` を使うわけではない。

どのディレクトリからファイルを返すかは、nginxの設定ファイルにある `root` ディレクティブで決まる。

```nginx
root /usr/share/nginx/html;
```

---

## 実際の環境で確認する

パスを暗記するより、設定ファイルを確認する方が実務的。

```bash
sudo grep -R "root " /etc/nginx
```

出力例。

```text
root         /usr/share/nginx/html;
```

実際にディレクトリが存在するか確認する。

```bash
ls -l /usr/share/nginx/html
```

---

## テスト用HTMLを配置する

```bash
echo "hello from private ec2" | sudo tee /usr/share/nginx/html/index.html
```

nginxを起動する。

```bash
sudo systemctl enable --now nginx
```

EC2内部からWebページを確認する。

```bash
curl http://localhost
```

期待する結果。

```text
hello from private ec2
```

---

## 学び

```text
Webページの保存場所
→ OSごとに完全固定されているわけではない

nginxのドキュメントルート
→ 設定ファイルのrootディレクティブで決まる

実務での確認方法
→ sudo grep -R "root " /etc/nginx
```

# ALBはなぜPrivate Subnet内のEC2へ接続できるのか？

## 概要

インターネット向けALBは、外部から受け取った通信を、Private Subnet内のEC2へ送れる。

```text
インターネット
↓
ALB
↓
Private Subnet
└── EC2
```

---

## 結論

ALBとPrivate EC2が、同じVPC内に存在するため。

同じVPC内の通信は、Route Tableに最初からある `local` ルートを使う。

```text
ALB
↓
VPC内のlocalルート
↓
Private EC2
```

---

## Route Table

VPCを作成すると、以下のようなルートが自動的に存在する。

| Destination | Target |
|---|---|
| `10.0.0.0/16` | `local` |

意味は以下。

```text
宛先が10.0.0.0/16の範囲内
↓
同じVPC内で通信する
```

---

## Private Subnetは孤立したSubnetではない

Private Subnetは、VPC内からも通信できないSubnetではない。

インターネットへ直接出入りするためのルートを持たないSubnet。

```text
Private Subnet
=
インターネットへ直接出入りする経路を持たないSubnet
```

同じVPC内のリソースとは通信できる。

---

## 構成例

```text
VPC：10.0.0.0/16

├── Public Subnet：10.0.1.0/24
│   └── ALB
│
└── Private Subnet：10.0.2.0/24
    └── EC2
```

通信経路。

```text
インターネット
↓
ALB
↓
localルート
↓
Private EC2
```

---

## Security Groupも必要

Route Tableで通信経路が存在していても、Security Groupで拒否されていると通信できない。

Private EC2側のSecurity Groupでは、ALBからの通信を許可する。

```text
Private EC2用Security Group

Inbound
Type：HTTP
Port：80
Source：ALB用Security Group
```

---

## なぜSourceにALB用Security Groupを指定するのか？

IPアドレスを直接指定するより、ALBから来た通信だけを許可しやすい。

```text
インターネット
↓
ALB用Security Group
↓
Private EC2用Security Group
↓
EC2
```

Private EC2へ、インターネットから直接アクセスさせる必要はない。

---

## 一言でまとめる

```text
ALBがPrivate EC2へ接続できる理由
→ ALBとEC2が同じVPC内にいる
→ localルートで通信できる
→ EC2側のSecurity GroupでALBからの通信を許可する
```

---

# AWS X-Rayとは？

## 概要

AWS X-Rayは、アプリケーションの処理の流れを追跡するサービス。

```text
AWS X-Ray
=
アプリケーションの処理経路を追跡するサービス
```

---

## 名前で覚える

病院のX線撮影をイメージする。

```text
X線
→ 体の中を見る

AWS X-Ray
→ アプリケーションの内部を見る
```

---

## 何を確認できるのか？

アプリケーションが複数のサービスを使っているとする。

```text
ユーザー
↓
ALB
↓
EC2
↓
Lambda
↓
DynamoDB
```

ページ表示が遅い場合、どこで時間がかかっているのか確認したい。

X-Rayを使うと、処理の流れを追跡し、遅延やエラーの発生箇所を確認しやすくなる。

---

## 例

```text
ユーザー
↓ 0.1秒
ALB
↓ 0.2秒
EC2
↓ 3.0秒
RDS
```

この場合、RDSへの問い合わせで時間がかかっている可能性がある。

---

## トレースとは？

トレースは、1つのリクエストがアプリケーション内をどのように通ったかを記録したもの。

```text
トレース
=
1件のリクエストが通った処理経路の記録
```

---

## トレースマップ

X-Rayでは、サービス同士のつながりを視覚的に確認できる。

```text
ユーザー
↓
API Gateway
↓
Lambda
↓
DynamoDB
```

現在は、CloudWatchコンソール内のX-Rayトレースマップから確認できる。

---

# CloudWatch・CloudTrail・X-Rayの違い

| サービス | 主な役割 | 覚え方 |
|---|---|---|
| Amazon CloudWatch | メトリクスやログを使って状態を監視する | 何が起きた？ |
| AWS CloudTrail | AWS上で行われた操作を記録する | 誰が操作した？ |
| AWS X-Ray | アプリのリクエスト経路を追跡する | どこで遅くなった？ |

---

## 例

Webページの表示が遅いとする。

```text
CloudWatch
→ CPU使用率は高いか？
→ エラーログは出ているか？

CloudTrail
→ 誰かがSecurity Groupを変更したか？
→ 誰かがEC2を停止したか？

X-Ray
→ どの処理に時間がかかっているか？
→ RDS問い合わせが遅いのか？
→ Lambda処理が遅いのか？
```

---

## 一言で覚える

```text
CloudWatch
→ 何が起きた？

CloudTrail
→ 誰がやった？

X-Ray
→ どこで遅くなった？
```

# Amazon EventBridgeとは？

## 概要

Amazon EventBridgeは、ある出来事をきっかけに、別の処理へつなぐサービス。

```text
Amazon EventBridge
=
イベントを受け取り、
別の処理へ橋渡しするサービス
```

---

## 名前で覚える

```text
Event
→ 出来事

Bridge
→ 橋

EventBridge
→ 出来事を別の処理へ橋渡しする
```

---

## イベントとは？

イベントは、システム内で発生した出来事。

```text
S3へファイルが置かれた
EC2の状態が変わった
注文が登録された
ユーザーがログインした
決められた時刻になった
```

---

## 基本の流れ

```text
何かが起きた
↓
EventBridge
↓
条件に合うか判断
↓
指定した処理を実行
```

---

## 例1：EC2の状態変更を通知する

```text
EC2が停止した
↓
EventBridge
↓
SNS
↓
メールなどで通知
```

---

## 例2：イベントをLambdaへ渡す

```text
イベントが発生
↓
EventBridge
↓
Lambda
↓
自動処理
```

---

## 例3：決まった時刻に処理する

```text
毎日9時になった
↓
EventBridge Scheduler
↓
Lambda
↓
定期処理を実行
```

---

## EventBridgeとS3イベント通知

S3へファイルが置かれたことをきっかけにLambdaを実行する場合、S3イベント通知を直接利用する構成もある。

```text
S3
↓
S3イベント通知
↓
Lambda
```

EventBridgeを使うと、より多くのサービスや処理へつなぎやすくなる。

```text
S3などでイベント発生
↓
EventBridge
├── Lambda
├── SNS
├── SQS
└── Step Functions
```

---

## EventBridgeの特徴

```text
AWSサービスのイベントを受け取れる
独自アプリのイベントを受け取れる
SaaSアプリのイベントを受け取れる
条件に応じて処理を振り分けられる
Lambda、SNS、SQSなどと連携できる
スケジュール実行にも使える
```

---

## 疎結合との関係

EventBridgeを使うと、サービス同士を直接強く結び付けずに連携できる。

```text
サービスA
↓ イベント発行
EventBridge
↓
サービスB
```

サービスAは、サービスBの細かな実装を意識しなくてもよい。

```text
EventBridge
→ サービス同士の間に入る連絡係
```

---

## 一言でまとめる

```text
EventBridge
→ イベントを別の処理へ橋渡しする

EventBridge Scheduler
→ 決められた時刻に処理を起動する
```

# AWS Trusted Advisorとは？

## 概要

AWS Trusted Advisorは、AWS環境を確認し、改善案を提示するサービス。

```text
AWS Trusted Advisor
=
AWS環境をチェックし、
改善アドバイスを提示するサービス
```

---

## 健康診断で例える

```text
AWS環境
↓
Trusted Advisorがチェック
↓
改善できるポイントを提示
```

Trusted Advisorは、AWS環境の健康診断に近い。

---

## 主な確認観点

Trusted Advisorでは、主に以下の観点からAWS環境を確認する。

```text
コスト最適化
パフォーマンス
セキュリティ
耐障害性
サービス制限・Service Quotas
```

---

## 例

```text
使っていないリソースが残っている
→ コストを削減できる可能性がある

Security Groupで不要なポートを広く公開している
→ セキュリティを改善できる可能性がある

冗長化されていない構成がある
→ 耐障害性を改善できる可能性がある

Service Quotasへ近づいている
→ 上限緩和を検討する
```

---

## 注意点

Trusted Advisorで確認できるチェック項目は、AWS Supportプランによって異なる。

```text
利用中のSupportプラン
↓
確認できるTrusted Advisorの項目が変わる
```

Cloud Practitionerでは、まず以下を覚える。

```text
Trusted Advisor
→ AWS環境の改善案を提示する
→ コスト、性能、セキュリティなどを確認する
```

---

# ステートフルとは？

## 概要

ステートフルは、前回までの通信や状態を覚えていること。

```text
ステートフル
=
前回までの状態を覚えている
```

---

## 会話で例える

```text
A「こんにちは」

B「こんにちは」

A「さっきの話だけど」

B「昨日話していたAWSのことですね」
```

Bは、以前の会話を覚えている。

これがステートフル。

---

# ステートレスとは？

## 概要

ステートレスは、前回までの通信や状態を覚えていないこと。

```text
ステートレス
=
前回までの状態を覚えていない
=
毎回初対面
```

---

## 会話で例える

```text
A「こんにちは」

B「こんにちは」

A「さっきの話だけど」

B「何の話ですか？」
```

Bは、以前の会話を覚えていない。

これがステートレス。

---

# Security Groupはステートフル

## 概要

Security Groupは、ステートフルな通信制御。

```text
Security Group
=
ステートフル
```

---

## 例

EC2へのHTTPアクセスを許可するとする。

```text
ユーザー
↓ HTTPリクエスト
EC2
```

Security GroupでInboundを許可する。

```text
Inbound
TCP 80
Source：0.0.0.0/0
```

すると、EC2からユーザーへ返す通信は、自動的に許可される。

```text
行きの通信を許可
↓
戻りの通信も自動で許可
```

---

## 覚え方

```text
Security Group
→ 前回の通信を覚えている
→ 行きを許可すれば、戻りは自動で許可される
```

---

# Network ACLはステートレス

## 概要

Network ACLは、ステートレスな通信制御。

略して、`NACL` と呼ばれる。

```text
Network ACL
=
ステートレス
```

---

## 例

Network ACLは、行きと戻りを別々に判断する。

```text
行きの通信
→ 許可ルールが必要

戻りの通信
→ 別途、許可ルールが必要
```

---

## 覚え方

```text
Network ACL
→ 前回の通信を覚えていない
→ 行きと戻りをそれぞれ許可する必要がある
```

---

# Security GroupとNetwork ACLの違い

| 項目 | Security Group | Network ACL |
|---|---|---|
| 適用範囲 | EC2、RDS、ENIなど | Subnet |
| 状態管理 | ステートフル | ステートレス |
| 戻りの通信 | 自動で許可される | 別途ルールが必要 |
| 許可・拒否 | 許可ルールのみ | 許可ルールと拒否ルール |
| 役割 | リソース単位の門番 | Subnet単位の門番 |

---

## 建物で例える

```text
VPC
└── Subnet
    ├── Network ACL
    │   └── 敷地の入口にいる警備員
    │
    └── EC2
        └── Security Group
            └── 部屋の入口にいる警備員
```
