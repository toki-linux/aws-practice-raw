# RDS実践：データベースの前提知識

## 目的

Private EC2からAmazon RDS for MySQLへ接続する実践を始める前に、次の内容を整理する。

```text
RDSとは何か
データベースはなぜ必要か
なぜ表形式で整理するのか
情報の更新と追加をどう使い分けるか
RDSがマネージドサービスであるとはどういうことか
```
---

# 1. Amazon RDSとは

RDSは、AWSが提供するデータベース用サービス。

正式名称。

```text
Amazon Relational Database Service
```

EC2が、

```text
サーバーを借りるサービス
```

なのに対して、RDSは、

```text
データベースを使いやすい形で借りるサービス
```

というイメージ。

今回の実践では、RDS上でMySQLを使用する。

```text
Amazon RDS
└── MySQL
```

MySQLは、データベースを管理するためのソフトウェア。

---

# 2. データベースは何のためにあるのか

## 疑問

```text
コンピュータなら、
情報が整理されていなくても探せるのではないか？
```

## 答え

情報を探すこと自体は可能。

しかし、情報が増えると次の問題が起きる。

```text
必要な情報を探すのに時間がかかる
正確に更新するのが難しい
複数人の同時操作に対応しにくい
データの矛盾が起きやすい
```

データベースを使うと、

```text
大量の情報を速く探せる
必要な情報だけ取得できる
正確に更新できる
同時アクセスを整理できる
矛盾を減らせる
```

たとえば、在庫が1個しかない商品を2人が同時に購入しようとした場合。

```text
在庫：1個

Aさんが購入
Bさんも同時に購入
```

単純なファイル管理では、両方の注文を受け付けてしまう可能性がある。

データベースでは、同時更新を整理し、

```text
Aさんが購入
↓
在庫を1個から0個へ更新
↓
Bさんは購入できない
```

という処理を行える。

---

# 3. なぜ表形式で整理するのか

## 疑問

```text
整理された文字の羅列でもよいのではないか？
```

## 答え

保存するだけなら、文字の羅列でもよい。

例。

```text
1,toki,Osaka
2,sachi,Tokyo
```

ただし、この形式では、

```text
1番目の項目はID
2番目の項目は名前
3番目の項目は住所
```

というルールを別途理解しておく必要がある。

表形式にすると、各項目の意味を明確に固定できる。

```text
id | name  | city
1  | toki  | Osaka
2  | sachi | Tokyo
```

意味。

```text
id列
→ 識別番号

name列
→ 名前

city列
→ 住所
```

表形式にすると、コンピュータへ条件を指定しやすい。

例。

```text
cityがOsakaの人だけ探す
nameだけ取り出す
特定のidの住所だけ変更する
```

表は、人間が眺めるためだけに使うものではない。

```text
コンピュータが、
条件を指定してデータを扱いやすくするための構造
```

でもある。

# 4. 情報は追加するのか、更新するのか

## 疑問

```text
田中さんが大阪から東京へ引っ越した場合、
大阪の田中さんと東京の田中さんを
別々に追加してもよいのではないか？
```

## 答え

用途によって、更新と追加を使い分ける。

基本。

```text
現在の状態だけ知りたい
→ 更新する

過去の履歴も残したい
→ 追加する

現在値と履歴の両方が必要
→ 現在値は更新し、履歴は別に追加する
```

---

# 5. なぜ別人として追加しないのか

名前や住所だけでは、同一人物か判断できない。

例。

```text
田中 太郎・大阪
田中 太郎・東京
```

この2人は、

```text
同じ人が引っ越した
```

可能性もある。

一方で、

```text
偶然同姓同名の別人
```

の可能性もある。

そこで、名前や住所ではなく、識別番号で管理する。

```text
user_id | name | city
101     | 田中 | 東京
```

田中さん本人は、

```text
user_id = 101
```

のまま。

住所だけを大阪から東京へ更新する。

```text
user_id | name | city
101     | 田中 | 大阪
```

変更後。

```text
user_id | name | city
101     | 田中 | 東京
```

---

# 6. 過去の履歴も残したい場合

現在の住所だけでなく、過去の住所も残したい場合は、履歴用の表を別に作る。

```text
address_history

user_id | city | valid_from | valid_to
101     | 大阪 | 2024-01-01 | 2026-05-31
101     | 東京 | 2026-06-01 | NULL
```

これにより、

```text
現在の住所
過去の住所
住所が変わった時期
```

を確認できる。

`NULL` は、

```text
値が入っていない
```

という意味。

この例では、

```text
東京に住んでいる期間は、まだ終了していない
```

ことを表している。

# 7. データベースは多くの役割がまとまりすぎていないか

## 疑問

LinuxやAWSでは、

```text
役割を小さく分ける
↓
必要なものを組み合わせる
```

という考え方が多かった。

例。

```text
EC2
S3
IAM
CloudWatch
VPC
```

一方で、データベースは、

```text
保存
検索
更新
履歴
権限
障害対応
```

など、多くの役割がまとまっているように見える。

## 答え

その違和感は正しい。

ただし、外から見ると1つに見えるだけで、内部では役割が分かれている。

データベース管理ソフトの内部イメージ。

```text
DBMS
├── 保存
├── 検索
├── 更新
├── インデックス管理
├── 同時アクセス制御
├── ログ
├── 障害復旧
└── 権限管理
```

つまり、

```text
全部を雑に1つで処理している
```

わけではない。

内部では役割を分けつつ、利用者には使いやすい単位でまとめて提供している。

---

# 8. DBMSとは

DBMSは、次の略。

```text
Database Management System
```

意味。

```text
データベースを管理するソフトウェア
```

今回使用するMySQLもDBMSの1つ。

```text
MySQL
→ データを保存・検索・更新するための
   データベース管理ソフトウェア
```

整理すると、次のようになる。

```text
データ
→ 保存されている情報そのもの

データベース
→ 整理して保存された情報の集まり

DBMS
→ データベースを管理するソフトウェア

MySQL
→ DBMSの1つ
```

---

# 9. RDSはさらに多くの機能をまとめて提供する

RDSは、MySQLなどのDBMSを動かすだけではない。

データベースの運用に必要な機能も、扱いやすい形でまとめて提供する。

```text
Amazon RDS
├── MySQLなどのDBMS
├── バックアップ
├── パッチ適用
├── 監視
├── ストレージ管理
└── 障害対応
```

AWSでは、このように管理作業の一部をAWS側に任せられるサービスを、

```text
マネージドサービス
```

と呼ぶ。

---

# 10. EC2へMySQLを入れる場合との違い

MySQLは、EC2へ自分でインストールすることもできる。

```text
EC2
└── MySQL
```

この場合は、利用者が広い範囲を管理する。

```text
OSの管理
MySQLのインストール
MySQLのアップデート
バックアップ
監視
障害対応
```

RDSを使う場合。

```text
Amazon RDS
└── MySQL
```

AWSへ管理作業の一部を任せられる。

```text
自分で管理する範囲を減らし、
データベースを利用することに集中できる
```

---

# 11. 今回の実践で作る構成

今回の実践では、Private EC2からRDS for MySQLへ接続する。

```text
自分のPC
↓
Session Manager
↓
Private EC2
↓ TCP 3306
Amazon RDS for MySQL
```

役割分担。

```text
Private EC2
→ MySQLクライアントを使って、
   RDSへ命令を送る

Amazon RDS for MySQL
→ データを保存・検索・更新する
```

---

# 12. 今回の理解まとめ

```text
データベース
→ 大量の情報を、
   正確に保存・検索・更新するための仕組み

表
→ データの項目と意味を固定し、
   コンピュータが条件を指定して扱いやすくする構造

DBMS
→ データベースを管理するソフトウェア

MySQL
→ DBMSの1つ

Amazon RDS
→ DBMSと運用機能を、
   AWSが扱いやすい形で提供するマネージドサービス
```

外から見ると1つの大きなサービスに見えるが、内部では役割ごとに分かれている。

```text
外から見る
→ 1つのサービス

内部を見る
→ 小さな役割に分かれている
```

この考え方は、EC2やS3などのAWSサービスとも共通している。
