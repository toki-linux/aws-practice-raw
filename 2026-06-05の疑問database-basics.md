
# データベース基礎とAWSのデータベースサービス

## 概要

AWSには、目的に応じて複数のデータベースサービスがある。

最初に押さえるポイントは、以下の違い。

```text
RDS・Aurora
→ 表と表の関係を扱うリレーショナルデータベース

DynamoDB
→ キーを使って高速にデータを取り出すNoSQLデータベース

Redshift
→ 大量のデータを集計・分析するためのデータウェアハウス
```

---

# リレーショナルデータベースとは？

## 概要

リレーショナルデータベースは、一言でいうと以下のようなデータベース。

```text
表と表を関連付けて管理するデータベース
```

`Relational` の元になっている `Relation` には、以下のような意味がある。

```text
Relation
→ 関係
→ 関連
→ つながり
```

---

## テーブルとは？

テーブルは、データベースの中にある表。

Excelの1シートをイメージすると分かりやすい。

たとえば、ユーザー情報を保存するテーブルは以下のようになる。

| user_id | name | age |
|---:|---|---:|
| 1 | toki | 21 |
| 2 | sachi | 30 |

```text
テーブル
=
リレーショナルデータベースでデータを保存する表
=
行と列で構成される
```

---

## 行と列

テーブルは、行と列で構成される。

```text
行
→ 1件分のデータ

列
→ データの項目
```

先ほどのテーブルでは、以下のように考える。

```text
1 | toki | 21
→ tokiというユーザー1人分のデータ

name
→ 名前を保存する項目

age
→ 年齢を保存する項目
```

---

## 表と表を関連付けるとは？

たとえば、ユーザー情報と注文情報を管理するとする。

### usersテーブル

| user_id | name |
|---:|---|
| 1 | toki |
| 2 | sachi |

### ordersテーブル

| order_id | user_id | item |
|---:|---:|---|
| 101 | 1 | keyboard |
| 102 | 1 | mouse |
| 103 | 2 | monitor |

`orders` テーブルには、注文者の名前を毎回書かず、`user_id` を保存している。

```text
usersテーブル
↓ user_idで関連付ける
ordersテーブル
```

これにより、以下のような情報を取り出せる。

```text
tokiが注文した商品
→ keyboard
→ mouse
```

---

## なぜ関連付けるのか？

すべての情報を1つの表へ詰め込むと、同じデータを何度も書く必要がある。

たとえば、注文のたびに住所や電話番号まで保存すると、重複が増える。

```text
注文1
→ toki、住所、電話番号、keyboard

注文2
→ toki、住所、電話番号、mouse
```

表を分けて関連付けると、重複を減らしやすい。

```text
usersテーブル
→ ユーザー情報を保存

ordersテーブル
→ 注文情報を保存

user_id
→ 2つの表をつなぐ
```

---

# SQLとは？

## 概要

SQLは、`Structured Query Language` の略。

日本語では、構造化問い合わせ言語と呼ばれる。

一言でいうと、以下のようなもの。

```text
SQL
=
リレーショナルデータベースへ命令するための言語
```

---

## SQLでできること

SQLを使うと、データベースに対して以下のような命令を出せる。

```text
データを登録する
データを検索する
データを更新する
データを削除する
```

---

## SQLの例

### データを検索する

```sql
SELECT name, age
FROM users;
```

意味は以下。

```text
usersテーブルから
name列とage列を取得する
```

### 条件を付けて検索する

```sql
SELECT name
FROM users
WHERE age >= 20;
```

意味は以下。

```text
usersテーブルから
20歳以上のユーザー名を取得する
```

---

# SQLクエリとは？

## 概要

クエリは、データベースへの問い合わせや命令。

```text
SQLクエリ
=
SQLで書かれたデータベースへの命令
```

たとえば、以下もSQLクエリ。

```sql
SELECT *
FROM users;
```

---

# SQLクエリの最適化とは？

## 概要

SQLクエリの最適化とは、データベースへの問い合わせを、速く・無駄なく処理できるように改善すること。

```text
SQLクエリの最適化
=
データベースへの問い合わせを効率よくすること
```

目的は以下。

```text
応答速度を改善する
データベースの負荷を減らす
無駄な処理を減らす
```

---

## 図書館で例える

図書館で本を探す場面を考える。

```text
方法1
→ すべての本棚を1冊ずつ確認する

方法2
→ ジャンル、著者名、本の番号を使って探す
```

方法2の方が速い。

SQLも同じ。

不要なデータまで毎回すべて確認すると、処理に時間がかかる。

---

## よくある最適化方法

代表例は以下。

```text
インデックスを付ける
必要な列だけ取得する
不要な検索条件を減らす
大量データを一度に取得しすぎない
```

---

## 必要な列だけ取得する

以下のSQLでは、すべての列を取得する。

```sql
SELECT *
FROM users;
```

名前だけ必要なら、以下の方が目的に合っている。

```sql
SELECT name
FROM users;
```

```text
SELECT *
→ すべての列を取得する

SELECT name
→ 必要な列だけ取得する
```

---

# インデックスとは？

## 概要

インデックスは、データベースの索引。

```text
インデックス
=
データベースの検索を速くするための索引
```

---

## 本で例える

本の巻末にある索引を考える。

```text
索引なし
→ 最初のページから1ページずつ探す

索引あり
→ キーワードから該当ページを見つける
```

データベースでも同じように、検索に使う列へインデックスを付けると、目的のデータを探しやすくなる。

---

## 注意点

インデックスは、付ければ付けるほど良いわけではない。

インデックスにも保存領域が必要。

また、データを追加・更新する際には、インデックス側も更新する必要がある。

```text
インデックスのメリット
→ 検索が速くなる

インデックスの注意点
→ 保存領域を使う
→ 登録や更新の処理が増える
```

---

# トランザクションとは？

## 概要

トランザクションは、複数の処理を1セットとして扱う仕組み。

```text
トランザクション
=
複数の処理をひとまとまりとして扱う仕組み
```

---

## 銀行振込で例える

AさんからBさんへ1万円を振り込む場合を考える。

必要な処理は以下。

```text
1. Aさんの残高から1万円を減らす
2. Bさんの残高へ1万円を増やす
```

1だけ成功して、2が失敗すると困る。

```text
Aさんの残高
→ 1万円減った

Bさんの残高
→ 増えていない
```

このような状態を防ぐため、複数の処理を1セットとして扱う。

```text
すべて成功
→ 処理を確定する

途中で失敗
→ 処理を元に戻す
```

---

## 覚え方

```text
トランザクション
=
全部成功なら確定
=
途中で失敗したら元に戻す
```

RDSやAuroraなどのリレーショナルデータベースが得意とする処理。

DynamoDBにも、トランザクション機能は用意されている。

---

# Amazon RDSとは？

## 概要

Amazon RDSは、リレーショナルデータベースを簡単に構築・運用するためのAWSサービス。

```text
Amazon RDS
=
リレーショナルデータベースのマネージドサービス
```

---

## データベースエンジンを選ぶ

RDSでは、実際に動かすデータベースエンジンを選ぶ。

```text
Amazon RDS
├── MySQL
├── PostgreSQL
├── MariaDB
├── Oracle Database
├── Microsoft SQL Server
└── Db2
```

今回の実践では、MySQLを選択した。

```text
Amazon RDS
↓
MySQLを選択
↓
RDS for MySQL
```

---

# Amazon Auroraとは？

## 概要

Amazon Auroraは、AWSがクラウド向けに設計したリレーショナルデータベース。

MySQL互換とPostgreSQL互換がある。

```text
Amazon Aurora
=
AWSがクラウド向けに設計した
MySQL・PostgreSQL互換のリレーショナルデータベース
```

RDSの管理画面から作成・管理できる。

---

# クロスリージョンリードレプリカとは？

## 概要

クロスリージョンリードレプリカは、別のリージョンへ作成する読み取り用のコピーDB。

```text
Cross-Region Read Replica
=
別リージョンへ作る読み取り用コピー
```

---

## 構成例

東京リージョンに本番RDSがあるとする。

```text
東京リージョン
└── RDS本体
```

シンガポールリージョンへRead Replicaを作成する。

```text
東京リージョン
└── RDS本体
    ↓ データを複製
シンガポールリージョン
└── Read Replica
```

---

## 何に使うのか？

主な用途は以下。

```text
読み取り処理の負荷を分散する
遠い地域のユーザーが近いリージョンから読み取れるようにする
災害対策として別リージョンへコピーを置く
```

---

## 注意点

Read Replicaは、基本的に読み取り用。

また、複製には遅延が発生する可能性がある。

```text
RDS本体
→ 書き込み可能

Read Replica
→ 基本的に読み取り用
→ 複製の遅延が発生する可能性がある
```

障害時には、Read Replicaを昇格させて、新しい書き込み先として利用できる。

ただし、単に作成しただけで自動的にすべて切り替わるわけではない。

---

# DynamoDBとは？

## 概要

Amazon DynamoDBは、AWSが提供するNoSQLデータベース。

キー・バリュー型、ドキュメント型のデータベースとして利用できる。

```text
DynamoDB
=
大量アクセスに対応しやすい
高速なNoSQLデータベース
```

---

## RDSとの違い

最初は、以下のように整理する。

```text
RDS
→ 表と表の関連付けを重視する
→ SQLで複雑な検索や集計ができる

DynamoDB
→ キーを使って高速にデータを取り出す
→ 大量アクセスへ対応しやすい
```

---

## 市役所と個人ファイルで例える

### RDS

```text
市役所の複数の台帳

住民票
戸籍
税金
```

必要なときに、複数の台帳を関連付けて確認する。

### DynamoDB

```text
個人ごとのファイル

tokiの情報
sachiの情報
```

キーを使って、必要なデータへ素早くアクセスする。

---

## DynamoDBのMulti-AZ構成

DynamoDBでは、ユーザーがAZを選んでテーブルを配置するわけではない。

テーブルを作成した時点で、AWS側が裏側で自動的に複数AZへデータを複製する。

標準では、リージョン内の3つのAZへ自動複製される。

```text
DynamoDBテーブルを作成
↓
AWS側が自動的に複数AZへ複製
↓
ユーザーが個別にAZを選ぶ必要はない
```

---

# すべてDynamoDB方式にできないのか？

## 結論

すべてのデータをDynamoDBへ保存すれば良いわけではない。

理由は、保存するデータの性質や、取り出し方が異なるため。

| サービス | 向いている用途 |
|---|---|
| RDS・Aurora | 表同士を関連付け、SQLで扱うデータ |
| DynamoDB | キーを使って高速に取得するデータ |
| S3 | ファイル、画像、ログ、バックアップ |
| EBS | EC2のディスク |
| Redshift | 大量データの分析 |

```text
すべて「データを保存するサービス」

ただし、
何を保存するか
どのように取り出すか
どのくらい更新するか
どのような検索をするか

が異なる
```

---

# Amazon Redshiftとは？

## 概要

Amazon Redshiftは、分析用のデータウェアハウス。

```text
Amazon Redshift
=
大量データを集計・分析するための
クラウドデータウェアハウス
```

---

## RDSとの違い

### RDS

アプリケーションが日々使うデータを扱う。

```text
会員登録
注文登録
在庫更新
予約受付
```

### Redshift

蓄積した大量のデータを分析する。

```text
月別の売上分析
顧客ごとの購買傾向
アクセスログ分析
経営レポート作成
```

---

## 覚え方

```text
日々の登録・更新
→ RDS・Aurora

大量データの集計・分析
→ Redshift
```

---
