# 課題1：セキュリティグループで80番ポートを閉じる

## 概要

AWS EC2上で動作しているWebサーバに対して、セキュリティグループのインバウンドルールからHTTP 80番を削除し、外部ブラウザからWebページにアクセスできない状態を再現しました。

その後、EC2内部ではnginxが正常に動作していることを確認し、原因がEC2内部ではなくAWS側の通信許可設定にあることを切り分けました。

---

## 症状

外部ブラウザからEC2のパブリックIPにアクセスしても、Webページが表示されませんでした。

---

## 原因

セキュリティグループのインバウンドルールで、HTTP 80番ポートが許可されていませんでした。

そのため、外部からEC2の80番ポートへ通信できない状態でした。

---

## 実施したこと

セキュリティグループのインバウンドルールから、HTTP 80番の許可設定を削除しました。

```text
インバウンドルール
HTTP
TCP
80
0.0.0.0/0
```

上記のHTTP許可ルールを削除し、外部からWebページにアクセスできない状態を再現しました。

---

## 切り分け

EC2内部ではnginxが起動していることを確認しました。

```bash
systemctl status nginx
```

80番ポートで待ち受けていることを確認しました。

```bash
ss -tulnp | grep :80
```

EC2内部からlocalhostへアクセスできることを確認しました。

```bash
curl http://localhost
```

この結果から、EC2内部のnginxやHTMLファイルの問題ではなく、AWS側の通信許可設定が原因だと判断しました。

---

## 課題1のポイント

課題1では、外部からはアクセスできませんが、EC2内部からの`localhost`アクセスは成功します。

```text
外部ブラウザ
→ アクセスできない

EC2内部 curl http://localhost
→ アクセスできる
```

この状態から、nginx自体は正常に動作しており、外部からEC2へ通信が届いていないと判断できます。

---

## 解決

セキュリティグループのインバウンドルールにHTTP 80番を再度追加しました。

```text
タイプ: HTTP
プロトコル: TCP
ポート範囲: 80
ソース: 0.0.0.0/0
```

設定後、外部ブラウザからEC2のパブリックIPへアクセスし、Webページが表示されることを確認しました。

---

## 学び

EC2でWebページが表示されない場合、nginxやHTMLファイルだけでなく、AWS側のセキュリティグループも確認する必要があります。

特に、EC2内部で`curl http://localhost`が成功する場合、Webサーバ自体は正常に動作している可能性が高いです。

その場合は、外部からEC2へ通信が届いているかを確認し、セキュリティグループやネットワーク設定を切り分けることが重要だと学びました。
