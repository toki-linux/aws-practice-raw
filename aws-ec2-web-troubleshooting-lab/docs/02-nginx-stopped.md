# 課題2：nginxを停止してWebページが表示されない状態を作る

## 概要

AWS EC2上で動作しているnginxを停止し、Webページが表示されない状態を再現しました。

その後、nginxの状態確認と復旧を行い、原因がAWS側のセキュリティグループではなく、EC2内部のWebサーバ停止であることを切り分けました。

---

## 症状

外部ブラウザからEC2のパブリックIPにアクセスしても、Webページが表示されませんでした。

---

## 原因

EC2内部でnginxサービスが停止していました。

そのため、EC2の80番ポートでWebリクエストを受け取るプロセスが存在せず、外部からアクセスしてもWebページを返せない状態でした。

---

## 実施したこと

nginxを停止しました。

```bash
sudo systemctl stop nginx
```

外部ブラウザからEC2のパブリックIPへアクセスし、Webページが表示されないことを確認しました。

---

## 切り分け

nginxの状態を確認しました。

```bash
systemctl status nginx
```

80番ポートで待ち受けているプロセスを確認しました。

```bash
ss -tulnp | grep :80
```

EC2内部からlocalhostへアクセスできるか確認しました。

```bash
curl http://localhost
```

課題1では`localhost`へのアクセスは成功しましたが、課題2ではnginx自体が停止しているため、`localhost`へのアクセスも失敗しました。

```text
課題1：外部アクセスNG、内部localhost OK
課題2：外部アクセスNG、内部localhost NG
```

この違いから、今回はAWS側のセキュリティグループではなく、EC2内部のnginx停止が原因だと判断しました。

---

## 課題2のポイント

セキュリティグループでHTTP 80番を許可していても、EC2内部でnginxが停止していればWebページは表示されません。

```text
外部ブラウザ
  ↓
EC2の80番へアクセス
  ↓
nginxが停止している
  ↓
応答できない
```

そのため、外部からアクセスできない場合は、AWS側の設定だけでなく、EC2内部のサービス状態も確認する必要があります。

---

## 解決

nginxを起動し直しました。

```bash
sudo systemctl start nginx
```

その後、nginxの状態を確認しました。

```bash
systemctl status nginx
```

80番ポートで待ち受けていることを確認しました。

```bash
ss -tulnp | grep :80
```

EC2内部からアクセスできることを確認しました。

```bash
curl http://localhost
```

最後に、外部ブラウザからEC2のパブリックIPへアクセスし、Webページが表示されることを確認しました。

---

## 学び

Webページが表示されない場合でも、原因はAWS側だけとは限りません。

セキュリティグループでHTTP 80番を許可していても、EC2内部のnginxが停止していればWebページは表示されません。

そのため、外部からアクセスできない場合は、AWS側の通信許可とEC2内部のサービス状態を分けて確認することが重要だと学びました。
