# 補足メモ：IGW・VPC Endpoint・DNS・Permission denied

## 1. Internet Gateway は、なぜ Subnet ではなく VPC にアタッチするのか

Internet Gateway は、VPC 全体とインターネットをつなぐ外部出入口。

```text
Internet
↓
Internet Gateway
↓
VPC
```

そのため、Internet Gateway は Subnet ではなく VPC にアタッチする。

最初は、

```text
Public Subnet
→ Internet Gatewayが付いている

Private Subnet
→ Internet Gatewayが付いていない
```

というイメージを持っていた。

しかし、正確には違う。

Internet Gateway は VPC に付いている。
Public Subnet と Private Subnet の違いは、Subnet が使用する Route Table にある。

### Public Subnet

```text
Route Table
├── 10.0.0.0/16 → local
└── 0.0.0.0/0   → Internet Gateway
```

Internet Gateway までのルートがある。

### Private Subnet

```text
Route Table
└── 10.0.0.0/16 → local
```

Internet Gateway までのルートがない。

### 一文で整理

```text
Internet Gateway
→ VPC全体の外部出入口

Public Subnet / Private Subnet
→ Subnetが使用するRoute Tableに、
  IGW向けルートがあるかどうかで決まる
```

---

## 2. VPC Endpoint は、どこに道を作るのか

VPC Endpoint は、Private EC2 から AWS サービスへ、インターネットを経由せずに通信するための仕組み。

```text
Private EC2
↓
VPC Endpoint
↓
AWSサービス
```

ただし、VPC Endpoint には複数の種類がある。

今回使用したのは次の2つ。

```text
Interface Endpoint
Gateway Endpoint
```

どちらも AWS サービスへ向かう経路を作るが、仕組みが異なる。

---

## 3. Interface Endpoint とは

Interface Endpoint は、Subnet 内に Private IP 付きの接続口を作る方式。

```text
Private Subnet
├── EC2
│
└── Interface Endpoint
    └── Endpoint用ENI
        └── Private IP
```

EC2 は、Endpoint 用 ENI の Private IP へ通信する。

```text
Private EC2
↓ HTTPS / TCP 443
Endpoint用ENI
↓
AWSサービス
```

今回使用した例:

```text
com.amazonaws.ap-northeast-1.ssm
com.amazonaws.ap-northeast-1.ssmmessages
com.amazonaws.ap-northeast-1.ec2messages
```

### 特徴

```text
Subnetを指定する
ENIが作成される
Private IPを持つ
Security Groupを付ける
Private DNSが重要
```

### イメージ

```text
Interface Endpoint
→ AWSサービスへつながるPrivateな受付窓口
```

---

## 4. Gateway Endpoint とは

Gateway Endpoint は、Route Table に AWS サービス向けの専用ルートを追加する方式。

```text
Private EC2
↓
Route Table
↓
S3向け通信だけ専用ルートへ流す
↓
S3 Gateway Endpoint
↓
Amazon S3
```

今回使用した例:

```text
com.amazonaws.ap-northeast-1.s3
```

タイプ:

```text
Gateway
```

### 特徴

```text
Subnetを直接指定しない
ENIを作らない
Private IPを持たない
Security Groupを付けない
Route Tableと関連付ける
```

S3 Gateway Endpoint を作成すると、関連付けた Route Table に S3 向けルートが追加される。

例:

```text
Destination: pl-xxxxxxxx
Target: vpce-xxxxxxxx
```

### イメージ

```text
Gateway Endpoint
→ AWSサービスへ向かう専用道路
```

---

## 5. Interface Endpoint と Gateway Endpoint の違い

| 項目                 | Interface Endpoint | Gateway Endpoint   |
| ------------------ | ------------------ | ------------------ |
| 今回の例               | SSM 系サービス          | Amazon S3          |
| 接続方法               | Private IP 付きの受付窓口 | Route Table 上の専用道路 |
| Subnet 指定          | する                 | しない                |
| ENI                | 作成される              | 作成されない             |
| Private IP         | 持つ                 | 持たない               |
| Security Group     | 付ける                | 付けない               |
| Route Table との関連付け | 基本不要               | 必要                 |
| DNS                | Private DNS が重要    | Route Table が重要    |

覚え方:

```text
Interface Endpoint
→ Private IP付きの受付窓口

Gateway Endpoint
→ Route Tableに追加する専用道路
```

## 6. VPC Endpoint の種類は、どのように選ぶのか

利用者が自由に好きな方式を選ぶわけではない。

まず、接続したい AWS サービスが、どの Endpoint 方式に対応しているかを確認する。

今回の例:

```text
SSM系サービス
→ Interface Endpoint

Amazon S3
→ Gateway Endpointを利用できる
```

S3 では Interface Endpoint も利用できるが、同じ VPC 内の Private EC2 から S3 へアクセスするだけなら、Gateway Endpoint がシンプル。

### 基本的な考え方

```text
S3 または DynamoDB
→ まず Gateway Endpoint を検討する

それ以外の多くのAWSサービス
→ Interface Endpointに対応しているか確認する
```

## 7. なぜ EC2 は IP アドレスではなくドメイン名で通信するのか

SSM Agent は、AWS サービスの IP アドレスを直接指定して通信するわけではない。

次のような AWS サービス名を使う。

```text
ssm.ap-northeast-1.amazonaws.com
ssmmessages.ap-northeast-1.amazonaws.com
ec2messages.ap-northeast-1.amazonaws.com
```

このサービス名を DNS が IP アドレスへ変換する。

```text
AWSサービス名
↓
DNSで名前解決
↓
Interface Endpoint用ENIのPrivate IP
```

### なぜ最初から IP アドレスを指定しないのか

IP アドレスは変更される可能性がある。

例:

```text
旧Endpoint用ENI
└── Private IP: 10.0.1.50

Endpointを作り直す
↓
新Endpoint用ENI
└── Private IP: 10.0.1.80
```

最初から IP アドレスを固定していると、変更のたびに接続設定を書き換える必要がある。

ドメイン名を使えば、DNS がその時点で正しい IP アドレスを返してくれる。

```text
ssm.ap-northeast-1.amazonaws.com
↓
DNS
↓
現在使用するEndpoint用ENIのPrivate IP
```

利用する側の SSM Agent は、同じサービス名を使い続けられる。

### 一文で整理

```text
ドメイン名を使う理由
→ 人間が分かりやすいだけではない
→ IPアドレスの変更へ柔軟に対応できる
```

## 8. S3 から EC2 へファイルを送るときに `Permission denied`

### 症状

S3 から EC2 へファイルをコピーしようとした。

```bash
aws s3 cp s3://バケット名/test.txt .
```

しかし、以下のエラーが表示された。

```text
Permission denied
```

### 原因

S3 の権限や VPC Endpoint が原因ではなかった。

コマンド実行時に、一般ユーザーが書き込めないディレクトリにいた。

今回は `/usr/bin` にいた。

```text
/usr/bin
→ Linuxのコマンドなどを置く場所
→ 一般ユーザーが自由にファイルを保存する場所ではない
```

コマンド末尾の `.` は、現在いるディレクトリを意味する。

```bash
aws s3 cp s3://バケット名/test.txt .
```

つまり、

```text
S3上のtest.txt
↓
現在いる /usr/bin に保存しようとした
↓
一般ユーザーには書き込み権限がない
↓
Permission denied
```

となった。

### 切り分け

現在地を確認する。

```bash
pwd
```

説明:

```text
現在いるディレクトリを表示する
```

### 解決

ホームディレクトリへ移動する。

```bash
cd ~
```

説明:

```text
現在のユーザーが自由に書き込みやすい
ホームディレクトリへ移動する
```

もう一度コピーする。

```bash
aws s3 cp s3://バケット名/test.txt .
```

ファイルを確認する。

```bash
ls -l
cat test.txt
```

### 学び

```text
Permission denied
→ 必ずしもIAM Policyの問題ではない

確認するもの
├── IAM Policy
├── VPC Endpoint
├── Route Table
└── Linux側の保存先とファイル権限
```

---

## 9. 一番重要な整理

```text
Internet Gateway
→ VPC全体の外部出入口

Route Table
→ 通信の宛先ごとに、次にどこへ送るか決める

Interface Endpoint
→ Subnet内にPrivate IP付きの受付窓口を作る

Gateway Endpoint
→ Route TableにAWSサービス向け専用道路を追加する

DNS
→ ドメイン名を、その時点で正しいIPアドレスへ変換する

IAM Policy
→ AWSサービスで何を操作してよいか決める

Permission denied
→ AWS側だけでなく、Linux側の権限も確認する
```
