# AMIとUser Dataの使い分け

## 目的

このドキュメントでは、AMIとUser Dataの違いと使い分けを整理します。

今回の実践では、nginxとWebページを設定済みのEC2をAMI化しました。  
以前の実践では、User Dataを使ってEC2起動時に初期構築を自動化しました。

この2つの違いを理解することが目的です。

---

## 一言での違い

```text
AMI
→ 完成済みの状態を保存して使う

User Data
→ 起動時にスクリプトで構築する
```

さらに短く言うと以下です。

```text
AMIは「完成状態の保存」
User Dataは「起動時の自動作業」
```

---

## AMIとは

AMIは、EC2インスタンスを起動するための元になるイメージです。

今回の場合、元EC2には以下が入っていました。

```text
Ubuntu
nginx
/var/www/html/index.html
```

この状態をAMIとして保存しました。

そのAMIから新しいEC2を起動すると、最初からnginxとindex.htmlが入った状態で起動します。

---

## User Dataとは

User Dataは、EC2の初回起動時に実行できるスクリプトです。

たとえば、以下のような処理を起動時に自動実行できます。

```text
apt update
nginxインストール
index.html作成
サービス起動
```

User Dataは、EC2を起動してから自動でセットアップする仕組みです。

---

## AMIのイメージ

```text
元EC2
├── Ubuntu
├── nginx
└── index.html

↓ AMI作成

自作AMI
└── 構築済み状態を保存

↓ AMIから起動

新EC2
├── Ubuntu
├── nginx
└── index.html
```

AMIは、構築済みの状態を保存して、同じ状態のEC2を作るイメージです。

---

## User Dataのイメージ

```text
新EC2起動
↓
User Data実行
↓
apt update
↓
nginxインストール
↓
index.html作成
↓
Webサーバ完成
```

User Dataは、起動時にスクリプトで構築するイメージです。

---

## AMIに向いているもの

AMIは、毎回変わらない共通設定を保存しておくのに向いています。

例です。

```text
OS基本設定
よく使うパッケージ
nginx
監視エージェント
共通設定ファイル
アプリのベース環境
```

AMIに入れておくと、毎回同じ構築作業をしなくて済みます。

---

## User Dataに向いているもの

User Dataは、起動時に変えたいものや、環境ごとに違う処理に向いています。

例です。

```text
サーバ名
環境名
最新アプリコードの取得
インスタンスごとに異なる設定
起動時だけ実行したい処理
```

起動するたびに違う内容を反映したい場合は、User Dataが向いています。

---

## 使い分けの基準

基本的な判断基準は以下です。

```text
変わらないもの
→ AMIに入れる

起動時に変えたいもの
→ User Dataで入れる
```

例です。

```text
nginxを毎回インストールするのが面倒
→ AMIに入れる

起動するたびに表示文言を変えたい
→ User Dataでindex.htmlを書く

全サーバ共通で入れるツールがある
→ AMIに入れる

環境ごとに違う設定がある
→ User Dataで設定する
```

---

## 実務っぽい使い方

実務では、AMIとUser Dataを組み合わせることがあります。

```text
AMI
→ ベース環境を作っておく

User Data
→ 起動時に環境ごとの仕上げをする
```

イメージです。

```text
共通AMI
├── Ubuntu
├── nginx
├── 基本ツール
└── 監視エージェント

User Data
├── この環境用の設定を書く
├── 最新アプリを取得する
└── サービスを起動する
```

---

## AMIのメリット

AMIのメリットは以下です。

```text
同じ状態のEC2を素早く作れる
毎回インストール作業をしなくてよい
起動後すぐに使える
障害時に同じ構成で復元しやすい
```

今回の実践では、AMIから起動した新EC2で、すぐに同じWebページが表示されました。

---

## User Dataのメリット

User Dataのメリットは以下です。

```text
起動時に自動構築できる
スクリプトで内容を変更しやすい
環境ごとの設定を反映しやすい
最新パッケージや最新コードを取得しやすい
```

AMIに固定せず、起動時に処理を変えたい場合に便利です。

---

## AMIの注意点

AMIは作成時点の状態を保存します。

そのため、AMI作成後に元EC2を変更しても、作成済みAMIには自動反映されません。

```text
AMI作成
↓
その時点の状態を保存

AMI作成後に元EC2を変更
↓
既存AMIには反映されない
```

変更後の状態をAMIにしたい場合は、新しくAMIを作成する必要があります。

---

## User Dataの注意点

User Dataは、基本的に初回起動時に実行されます。

また、スクリプトの内容に問題があると、起動後の構築に失敗することがあります。

```text
User Dataのスクリプトが正しい
→ 起動時に自動構築できる

User Dataのスクリプトにミスがある
→ 起動後に想定通り構築されない
```

---

## 今回の実践での違い

今回のAMI実践では、元EC2でnginxとindex.htmlを作成してからAMI化しました。

```text
AMI実践
→ すでにnginxとindex.htmlが入った状態を保存
```

User Data実践では、EC2起動時にnginxインストールやindex.html作成を自動実行しました。

```text
User Data実践
→ 起動時にスクリプトでnginxとindex.htmlを作成
```

---

## まとめ

```text
AMI
→ 作ったものを保存して再利用する

User Data
→ 起動時に自動で作る
```

今回の理解としては以下です。

```text
User Data
→ 0から作る自動化

AMI
→ 作ったものを保存して再利用
```

どちらか一方だけではなく、目的に応じて使い分けたり、組み合わせたりすることが重要です。
