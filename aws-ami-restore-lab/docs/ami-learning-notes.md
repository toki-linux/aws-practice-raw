# AMI実践：学びのまとめ

## 今回学んだこと

今回の実践では、設定済みEC2をAMIとして保存し、そのAMIから新しいEC2を起動しました。

結果として、新しいEC2でも元EC2と同じWebページを表示できました。

今回の一番大事な学びは以下です。

```text
AMIはEC2の起動元イメージ。
公式Ubuntu AMIの代わりに、自分で作ったAMIを使ってEC2を起動できる。
```

---

## AMIとは

AMIは、EC2インスタンスを起動するための元になるイメージです。

EC2作成時に選ぶUbuntuやAmazon Linuxも、AMIです。

```text
Ubuntu AMI
→ Ubuntuが入ったEC2を起動するための公式イメージ

自作AMI
→ 自分で設定した状態のEC2を起動するためのカスタムイメージ
```

---

## 今回作成したAMI

今回作成したAMIには、元EC2の状態が含まれています。

主に以下が含まれています。

```text
Ubuntu
nginx
/var/www/html/index.html
インストール済みパッケージ
設定ファイル
```

そのため、AMIから新しいEC2を起動すると、nginxとWebページが入った状態で起動しました。

---

## 公式Ubuntu AMIと自作AMIの違い

今までEC2作成時に選んでいたUbuntuは、公式が用意しているAMIです。

```text
公式Ubuntu AMI
→ まっさらなUbuntuから始める

自作AMI
→ 自分で設定済みの状態から始める
```

今回の自作AMIでは、nginxとindex.htmlがすでに入っている状態から起動できました。

---

## AMIとスナップショットの関係

AMIを作成すると、裏側でEBSスナップショットが作成されます。

```text
EC2
↓
AMI作成
↓
AMIが登録される
↓
EBSスナップショットが作成される
```

AMIはEC2を起動するためのテンプレートです。  
スナップショットはEBSボリュームのバックアップです。

---

## AMIとスナップショットの違い

```text
AMI
→ EC2を起動するためのテンプレート

スナップショット
→ EBSボリュームのバックアップ
```

AMIだけではなく、その裏側にあるスナップショットも意識する必要があります。

特に削除時は、AMIの登録解除だけでなく、スナップショット削除も確認します。

---

## AMIに含まれるもの

AMIには、主にルートEBSの中身が含まれます。

今回の場合は以下です。

```text
OS
nginx
index.html
インストール済みパッケージ
設定ファイル
```

つまり、EC2の「中身」がAMIとして保存されます。

---

## AMIに含まれないもの

AMIからEC2を起動しても、以下は改めて選びます。

```text
インスタンスタイプ
キーペア
セキュリティグループ
パブリックIP
サブネット
IAM Role
```

これらはEC2を起動するときに指定するAWS側の設定です。

---

## インスタンスタイプが引き継がれない理由

AMIはEC2の中身の元ネタです。

一方で、インスタンスタイプはAWS側で選ぶサーバの性能・サイズです。

```text
AMI
→ インスタンスの中身

インスタンスタイプ
→ CPU・メモリなどの性能
```

そのため、同じAMIでも、別のインスタンスタイプで起動できます。

```text
同じAMI
→ t2.microで起動
→ t3.microで起動
→ t3.smallで起動
```

---

## キーペアが引き継がれない理由

キーペアは、EC2にSSH接続するための鍵です。

AMIに固定されるものではなく、新しいEC2を起動するときに選びます。

```text
AMI
→ 中身の元ネタ

キーペア
→ そのEC2に入るための鍵
```

そのため、AMIから起動する場合でもキーペアは改めて選びます。

---

## セキュリティグループが引き継がれない理由

セキュリティグループは、EC2に対する通信ルールです。

これもAMIに固定されるものではなく、EC2起動時に設定します。

```text
AMI
→ EC2の中身

セキュリティグループ
→ 通信ルール
```

今回も、新EC2起動時にSSHとHTTPを許可するセキュリティグループを設定しました。

---

## AMIとLaunch Templateの違い

AMIはEC2の中身を保存するものです。

インスタンスタイプ、キーペア、セキュリティグループなどの起動設定までまとめたい場合は、Launch Templateという考え方があります。

```text
AMI
→ EC2の中身を保存する

Launch Template
→ AMI、インスタンスタイプ、キーペア、セキュリティグループなど起動設定をまとめる
```

今回の範囲ではLaunch Templateは使っていませんが、AMIだけでは起動時の全設定が固定されるわけではないことを理解しました。

---

## AMIを使うメリット

AMIを使うメリットは以下です。

```text
同じ状態のEC2をすぐ作れる
毎回同じセットアップをしなくて済む
障害時に復元しやすい
複数台の同じ構成のサーバを作りやすい
```

今回、AMIから新EC2を起動した時点で、nginxとindex.htmlが入っていました。

---

## AMIの注意点

AMIは作成時点の状態を保存します。

元EC2を後から変更しても、作成済みAMIには自動反映されません。

```text
AMI作成時点
→ その時点の状態を保存

AMI作成後の変更
→ 既存AMIには反映されない
```

変更後の状態を再利用したい場合は、新しくAMIを作り直す必要があります。

---

## 削除時の注意

AMI実践では、削除時に以下を確認しました。

```text
元EC2
AMIから作成した新EC2
AMI
スナップショット
EBSボリューム
```

特に重要なのは以下です。

```text
AMIを登録解除しても、関連スナップショットが残ることがある
```

そのため、AMI登録解除後にスナップショットも削除しました。

---

## 今回の理解まとめ

```text
AMI
→ EC2の起動元イメージ

公式Ubuntu AMI
→ まっさらなUbuntuを起動する元ネタ

自作AMI
→ 自分で設定済みのEC2を起動する元ネタ

スナップショット
→ AMI作成時に裏側で作られるEBSのバックアップ

インスタンスタイプ
→ AWS側で選ぶサーバ性能

キーペア
→ SSH接続用の鍵

セキュリティグループ
→ 通信ルール
```

---

## 一番重要なポイント

```text
AMIは「EC2の中身」を保存するもの。
インスタンスタイプやキーペア、セキュリティグループは、EC2起動時に改めて選ぶAWS側の設定。
```

今回の実践で、AMIを使って設定済みEC2を再作成できることを確認できました。
