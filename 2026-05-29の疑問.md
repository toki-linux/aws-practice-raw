# SSM・ENI・DNS・CloudWatch Agent の概念整理

## 概要

Private Subnet に配置した EC2 インスタンスへ、AWS Systems Manager Session Manager を使って接続した。

さらに、Systems Manager Run Command を使って CloudWatch Agent をインストールし、EC2 内のログを CloudWatch Logs へ送信した。

このドキュメントでは、構築中に疑問に感じた以下の点を整理する。

* ENI とは何か
* なぜ Interface VPC Endpoint は Subnet 内に存在するのか
* EC2 用 Security Group の Inbound Rule が空でも通信できる理由
* なぜ Private DNS を有効にするのか
* IAM Role と IAM Policy の関係
* SSM Agent と CloudWatch Agent の違い
* Run Command が何をしているのか
* `/opt` ディレクトリの役割

---

# 1. 全体構成

今回の構成は以下。

```text
AWS Cloud
└── VPC
    └── Private Subnet
        ├── EC2
        │   ├── ENI
        │   ├── SSM Agent
        │   └── CloudWatch Agent
        │
        └── SSM系 Interface VPC Endpoint
            └── Endpoint用 ENI
                ├── Private IP
                └── Security Group
```

Private EC2 には Public IP を付与していない。

また、Security Group で SSH の TCP 22 番も開放していない。

```text
Public IP: なし
SSH: 開放しない
Inbound Rule: なし
```

EC2 への接続には、Session Manager を使用した。

---

# 2. ENI とは何か

ENI は、以下の略。

```text
Elastic Network Interface
```

ざっくり言うと、

```text
ENI
= AWS上の仮想ネットワークカード
```

EC2 インスタンスは、ENI を通して VPC 内のネットワークに参加する。

```text
EC2
└── ENI
    ├── Private IP
    ├── Security Group
    └── Subnetに所属
```

つまり、EC2 本体が直接ネットワークへ接続しているというより、

```text
EC2
↓
ENIというネットワーク接続口
↓
VPC内のネットワーク
```

という形で通信する。

---

# 3. なぜ VPC Endpoint が Subnet 内に存在するのか

`VPC Endpoint` という名前を見ると、

```text
VPC全体に存在する抽象的な仕組み
```

のように感じる。

ただし、今回使用した **Interface VPC Endpoint** には、実際の接続口が必要。

Interface VPC Endpoint を作成すると、指定した Subnet 内に Endpoint 用 ENI が作成される。

```text
VPC
└── Private Subnet
    ├── EC2
    │   └── EC2用ENI
    │       └── Private IP: 10.0.1.10
    │
    └── Interface VPC Endpoint
        └── Endpoint用ENI
            └── Private IP: 10.0.1.50
```

EC2 から見ると、

```text
同じVPC内に、
AWSサービスへつながるPrivateな接続口が作られた
```

という状態。

## 管理上の名前と実体

```text
VPC Endpoint
→ AWSサービスへ接続する仕組み全体の名前

Endpoint用ENI
→ VPC内に作られる実際の接続口
```

この2つは矛盾しない。

## たとえ

```text
VPC Endpoint
→ 会社に導入した内線電話サービス

Endpoint用ENI
→ 各部屋に置かれた実際の電話機
```

サービス全体は会社に導入されるが、実際に通話するためには電話機が必要。

---

# 4. Interface VPC Endpoint と Gateway VPC Endpoint

VPC Endpoint には複数の種類がある。

今回使用したのは、主に次の2種類。

```text
Interface VPC Endpoint
Gateway VPC Endpoint
```

---

## 4.1 Interface VPC Endpoint

Interface VPC Endpoint は、Subnet 内に Private IP を持つ接続口を作る方式。

```text
Private EC2
↓
Endpoint用ENIのPrivate IP
↓
AWSサービス
```

特徴:

```text
Subnetを指定する
ENIが作成される
Private IPを持つ
Security Groupを付ける
Private DNSが重要
```

今回使用した例:

```text
com.amazonaws.ap-northeast-1.ssm
com.amazonaws.ap-northeast-1.ssmmessages
com.amazonaws.ap-northeast-1.ec2messages
```

---

## 4.2 Gateway VPC Endpoint

Gateway VPC Endpoint は、Route Table に AWS サービス向けの専用ルートを追加する方式。

```text
Private EC2
↓
Route Table
↓
S3向け通信だけ専用ルートへ流す
↓
S3 Gateway Endpoint
↓
Amazon S3
```

特徴:

```text
Subnetを直接指定しない
ENIを作らない
Private IPを持たない
Security Groupを付けない
Route Tableと関連付ける
```

今回使用した例:

```text
com.amazonaws.ap-northeast-1.s3
```

タイプ:

```text
Gateway
```

---

## 4.3 違いの整理

| 項目                 | Interface VPC Endpoint | Gateway VPC Endpoint |
| ------------------ | ---------------------- | -------------------- |
| 今回の用途              | SSM 系サービス              | Amazon S3            |
| Subnet 指定          | する                     | しない                  |
| ENI                | 作成される                  | 作成されない               |
| Private IP         | 持つ                     | 持たない                 |
| Security Group     | 付ける                    | 付けない                 |
| Route Table との関連付け | 基本不要                   | 必要                   |
| DNS                | Private DNS が重要        | Route Table が重要      |
| イメージ               | Private な受付窓口          | 専用道路                 |

覚え方:

```text
Interface Endpoint
→ Private IP付きの受付窓口

Gateway Endpoint
→ AWSサービス専用の道路
```

# 5. Security Group を2つ作る理由

今回、Security Group を2つ作成した。

```text
EC2用 Security Group
Endpoint用 Security Group
```

---

## 5.1 EC2 用 Security Group

EC2 用 Security Group は以下の設定でよい。

```text
Inbound:
なし

Outbound:
すべて許可
```

SSH の TCP 22 番を開ける必要もない。

HTTP の TCP 80 番を開ける必要もない。

理由は、外部から EC2 へ直接アクセスしないため。

```text
自分のPC
×→ EC2へ直接SSH

ブラウザ
×→ EC2へ直接HTTPアクセス
```

EC2 は、自分から Interface Endpoint へ HTTPS 通信を送信する。

```text
Private EC2
↓ HTTPS / TCP 443
Interface Endpoint
```

---

## 5.2 Endpoint 用 Security Group

Endpoint 用 Security Group では、以下を許可する。

```text
Inbound:
HTTPS / TCP 443
Source: EC2用 Security Group
```

意味:

```text
EC2用Security Groupが付いたEC2から、
Endpoint用ENIへ届くHTTPS通信を許可する
```

Security Group 自体が通信を送信するわけではない。

```text
EC2
→ 通信する主体

Endpoint用ENI
→ 通信を受け取る接続口

Security Group
→ 通信を通してよいか判断する門
```

---

# 6. EC2 の Inbound Rule が空でも応答が戻る理由

疑問:

```text
EC2用Security GroupのInbound Ruleが空なら、
EndpointからEC2へ応答を返せないのではないか？
```

答え:

```text
Security Groupはステートフルだから
```

Security Group は、許可された通信に対する戻り通信を自動的に許可する。

```text
EC2
↓ 自分からHTTPS通信を開始
Interface Endpoint
↓ 応答
EC2へ戻る
```

EC2 から開始した通信への返答であれば、EC2 用 Security Group の Inbound Rule に個別で書く必要はない。

## ステートフルの意味

```text
行きで許可された通信
↓
その通信に対する戻り通信
↓
自動的に許可される
```

ただし、相手側から新しく始まる通信は別扱い。

```text
相手側
↓ 新しい通信を開始
EC2
```

この場合は、EC2 用 Security Group の Inbound Rule で明示的な許可が必要。

## 補足

ステートフルなのは EC2 そのものではない。

```text
誤:
EC2はステートフル

正:
Security Groupはステートフル
```

# 7. なぜ DNS を有効にする必要があるのか

SSM Agent は、AWS サービスの IP アドレスを直接指定して通信するわけではない。

AWS サービス名を使う。

例:

```text
ssm.ap-northeast-1.amazonaws.com
ssmmessages.ap-northeast-1.amazonaws.com
ec2messages.ap-northeast-1.amazonaws.com
```

しかし、Private EC2 はインターネットへ出られない。

そのため、通常の外向き接続先に名前解決されても到達できない。

そこで、Interface VPC Endpoint の Private DNS を有効にする。

```text
ssm.ap-northeast-1.amazonaws.com
↓ DNSで名前解決
Interface Endpoint用ENIのPrivate IP
```

これにより、SSM Agent は AWS サービス名をそのまま使用しながら、VPC 内の Endpoint へ通信できる。

## 通信の流れ

```text
SSM Agent
↓
「ssm.ap-northeast-1.amazonaws.com に接続したい」
↓
VPC内のDNS
↓
「Interface EndpointのPrivate IPへ行って」
↓
Endpoint用ENI
↓
AWS Systems Manager
```

## 一文で整理

```text
Private DNSを有効にする理由
= SSM Agentが使用するAWSサービス名を、
  インターネット側ではなく、
  Interface EndpointのPrivate IPへ向けるため
```

# 8. なぜ IP アドレスではなくドメイン名を使うのか

ドメイン名は、人間が覚えやすくするためだけに存在するわけではない。

IP アドレスの変更へ柔軟に対応する役割もある。

```text
サービス名
↓
DNS
↓
その時点で使うべきIPアドレス
```

もし接続先を IP アドレスで固定すると、IP が変わったときに接続設定を書き換える必要がある。

```text
旧IP: 10.0.1.50
↓
Endpointを再作成
↓
新IP: 10.0.1.80
```

ドメイン名を使えば、DNS 側の案内先だけを変更できる。

```text
ssm.ap-northeast-1.amazonaws.com
↓
DNS
↓
10.0.1.80
```

利用側の SSM Agent は、同じサービス名を使い続けられる。

# 9. DNS resolution と DNS hostnames

VPC には DNS 関連設定がある。

```text
DNS resolution
DNS hostnames
```

## DNS resolution

```text
AWSが用意したDNS機能を使って、
名前解決してよいか
```

## DNS hostnames

```text
EC2などへDNS名を付与できるようにするか
```

Interface VPC Endpoint の Private DNS を利用する場合は、VPC 側の DNS 設定と、Endpoint 側の Private DNS 設定を確認する。

```text
VPC
├── DNS resolution: 有効
└── DNS hostnames: 有効

Interface VPC Endpoint
└── Private DNS: 有効
```

DNS は Subnet に対して有効化するものではない。

```text
誤:
SubnetでDNSを有効化する

正:
VPCのDNS属性を確認する
+
Interface EndpointでPrivate DNSを有効化する
```

# 10. IAM Role と IAM Policy

EC2 には、必要な権限を持つ IAM Role を付ける。

```text
EC2
└── IAM Role
    ├── AmazonSSMManagedInstanceCore
    └── CloudWatchAgentServerPolicy
```

## IAM Role

```text
EC2へ権限を持たせるための入れ物
```

## IAM Policy

```text
何を許可するかを書いた権限定義
```

IAM Role に IAM Policy を付け、その Role を EC2 に付与する。

```text
IAM Policy
↓
IAM Role
↓
EC2
```

## ロールは1つでよい

EC2 に必要な機能ごとに Role を複数作る必要はない。

```text
誤:
SSM用Role
CloudWatch用Role
S3用Role
```

ではなく、

```text
正:
EC2用Role
├── SSM用Policy
├── CloudWatch用Policy
└── S3用Policy
```

と考える。

EC2 に同時に付ける IAM Role は、基本的に1つ。

その Role に必要な Policy を複数追加する。

# 11. SSM Agent とは

SSM Agent は、EC2 内で動く常駐プログラム。

```text
SSM Agent
= Systems Managerからの命令を受け取り、
  EC2内で処理する係
```

今回の Ubuntu や Amazon Linux など、一部の AMI では最初から導入されている場合がある。

Session Manager で EC2 を操作できるのは、EC2 内で SSM Agent が動いているため。

```text
AWS Systems Manager
↓
Interface VPC Endpoint
↓
SSM Agent
↓
EC2内で処理
```

確認コマンド:

```bash
systemctl list-units --type=service | grep -i ssm
```

説明:

```text
Linux上で認識されているサービス一覧から、
ssmという文字を含む行を表示する
```

---

# 12. CloudWatch Agent とは

CloudWatch Agent は、EC2 内のログやメトリクスを CloudWatch へ送る常駐プログラム。

```text
CloudWatch Agent
= EC2内のログやメトリクスを
  CloudWatchへ送る係
```

例:

```text
EC2内のログファイル
↓
CloudWatch Agent
↓
CloudWatch Logs
```

SSM Agent と CloudWatch Agent は別物。

```text
SSM Agent
→ AWSからEC2を操作するため

CloudWatch Agent
→ EC2からCloudWatchへ情報を送るため
```

## Agent は複数入れられる

EC2 へ入れられる Agent は1つだけではない。

```text
EC2
├── SSM Agent
├── CloudWatch Agent
└── その他のAgent
```

目的ごとに複数の Agent を利用できる。

---

# 13. なぜ CloudWatch Agent は後から入れるのか

SSM Agent は、AWS から EC2 を操作するための入口として使用する。

```text
Session Manager
Run Command
```

などに必要。

一方、CloudWatch Agent は、すべての EC2 で必要になるわけではない。

```text
ログをCloudWatch Logsへ送らない
別の監視ツールを使う
詳細メトリクスを送らない
```

という構成もある。

そのため、必要な場合だけ後から導入する。

```text
SSM Agent
→ 最初から入っていることがある

CloudWatch Agent
→ 必要に応じて後から追加する
```

# 14. Run Command とは

Run Command は、Systems Manager の機能の1つ。

ひとことで言うと、

```text
AWSコンソールからEC2へ命令を送り、
EC2内でコマンドや処理を実行させる機能
```

## イメージ

```text
AWSコンソール
↓
Systems Manager Run Command
↓
SSM Agent
↓
EC2内で処理を実行
```

重要なのは、Run Command 自体が EC2 内で直接処理するわけではないこと。

```text
Run Command
→ 命令を送る仕組み

SSM Agent
→ 命令を受け取り、EC2内で実行する係
```

## SSHとの違い

SSH を使う場合:

```text
自分
↓
SSHでEC2へログイン
↓
コマンドを手入力
```

Run Command を使う場合:

```text
自分
↓
AWSコンソールから命令
↓
SSM AgentがEC2内で処理
```

Run Command を使うと、EC2へ直接ログインせずに処理できる。

# 17. HTTP の Inbound Rule が不要だった理由

CloudWatch Logs の実践では、EC2 内で nginx を外部公開することが目的ではなかった。

目的は、

```text
EC2内のログ
↓
CloudWatch Agent
↓
CloudWatch Logs
```

を確認すること。

そのため、外部ブラウザから nginx へアクセスさせる必要はない。

```text
ブラウザ
×→ EC2のnginx
```

EC2 内から、自分自身へ `curl` すればログを作れる。

```bash
curl http://localhost
```

説明:

```text
EC2自身から、
EC2内で動くWebサーバーへHTTPアクセスする
```

この通信は EC2 内で完結する。

そのため、EC2 用 Security Group で HTTP の TCP 80 番を Inbound Rule に追加する必要はない。

```text
外部へWebページを公開する
→ HTTP 80番のInbound許可が必要

EC2内でログだけ作る
→ HTTP 80番のInbound許可は不要
```

# 18. CloudWatch Logs 実践の流れ

今回の流れは以下。

```text
1. Private EC2を作成
2. SSM Agentが動いていることを確認
3. Session ManagerでEC2へ接続

5. CloudWatch Agentをインストール
6. CloudWatch Agentの設定ファイルを作成
7. CloudWatch Agentを起動
8. EC2内のログファイルを更新
9. CloudWatch Logsで確認
```

# 20. 学び

今回の実践を通じて、Private EC2 から AWS サービスへ通信するためには、複数の要素が必要だと分かった。

```text
通信経路
→ VPC Endpoint

通信の門
→ Security Group

接続先への案内
→ DNS

AWSサービスを操作する権限
→ IAM Role / IAM Policy

EC2内でAWSの要求を処理する係
→ SSM Agent

ログをCloudWatchへ送る係
→ CloudWatch Agent
```

これらは似ているように見えるが、それぞれ役割が異なる。

一言でまとめると、

```text
Private EC2を安全に管理するには、
道・門・案内・権限・実行役が必要
```

ということ。
