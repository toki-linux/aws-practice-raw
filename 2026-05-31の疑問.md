# 補足メモ：Endpoint の選び方と残っている理解ポイント

## 1. Interface Endpoint と Gateway Endpoint は、どう選ぶのか

VPC Endpoint を作成するときは、最初から自由に方式を選ぶわけではない。

まず、

```text
どのAWSサービスへ接続したいか
```

を確認する。

そのうえで、

```text
そのAWSサービスが、
どのEndpoint方式に対応しているか
```

を見る。

---

## 2. 判断の順番

実務では、次の順番で考える。

```text
1. 接続先のAWSサービスは何か
2. そのサービスが対応しているEndpoint方式は何か
3. 通信元は同じVPC内だけか
4. オンプレミスや別VPCからも接続するか
5. 料金・Security Group・Route Tableの管理方法を比較する
```

---

## 3. 目的は共通している

Interface Endpoint と Gateway Endpoint は、仕組みは異なる。

ただし、目的は共通している。

```text
Private EC2
↓
Internet GatewayやNAT Gatewayを使わずに
↓
AWSサービスへ接続する
```

つまり、

```text
Privateな環境から、
AWSサービスへ専用経路で接続する
```

ために使用する。

---

## 4. Gateway Endpoint

Gateway Endpoint は、Route Table に専用ルートを追加する方式。

```text
Private EC2
↓
Route Table
↓
AWSサービス向け専用ルート
↓
Gateway Endpoint
↓
AWSサービス
```

代表的な対象サービス:

```text
Amazon S3
DynamoDB
```

特徴:

```text
S3・DynamoDB向け
Route Tableに専用ルートを追加する
ENIを作らない
Private IPを持たない
Security Groupを付けない
```

一文で整理すると、

```text
Gateway Endpoint
→ S3・DynamoDB向けのシンプルな専用ルート
```

---

## 5. Interface Endpoint

Interface Endpoint は、Subnet 内に Private IP 付きの接続口を作る方式。

```text
Private EC2
↓
Endpoint用ENIのPrivate IP
↓
AWSサービス
```

特徴:

```text
多くのAWSサービスで使用できる
Subnet内にENIを作る
Private IPを持つ
Security Groupを付ける
Private DNSが重要
```

一文で整理すると、

```text
Interface Endpoint
→ Private IP付きの入口を作る汎用的な方式
```

---

## 6. 今回の使い分け

今回の構成では、以下を使用した。

### SSM 系サービス

```text
com.amazonaws.ap-northeast-1.ssm
com.amazonaws.ap-northeast-1.ssmmessages
com.amazonaws.ap-northeast-1.ec2messages
```

方式:

```text
Interface Endpoint
```

理由:

```text
Private EC2からSystems Managerへ接続するため
```

---

### Amazon S3

```text
com.amazonaws.ap-northeast-1.s3
```

方式:

```text
Gateway Endpoint
```

理由:

```text
Private EC2からS3へ接続するため
```

---

## 7. S3 ではどちらを選ぶか

S3 は、Gateway Endpoint と Interface Endpoint の両方に対応している。

同じ VPC 内の Private EC2 から S3 へ接続するだけなら、まず Gateway Endpoint を検討する。

```text
同じVPC内のPrivate EC2
↓
S3へアクセス
↓
Gateway Endpoint
```

理由:

```text
構成がシンプル
Route Tableへの関連付けで利用できる
Security Group不要
追加料金なし
```

一方で、オンプレミスや別 VPC から接続したい場合は、Interface Endpoint を検討する。

```text
オンプレミス
または
別VPC
↓
Interface Endpoint
↓
S3
```

---

## 8. 覚え方

```text
S3・DynamoDB
→ まずGateway Endpointを検討する

それ以外の多くのAWSサービス
→ Interface Endpointに対応しているか確認する
```

# 修正して覚えるポイント

## 9. DNS 設定の場所

誤った理解:

```text
DNSはSubnetの設定
```

正しい理解:

```text
DNSは主にVPCとInterface Endpointの設定
```

確認する場所:

```text
VPC
├── DNS resolution
└── DNS hostnames

Interface Endpoint
└── Private DNS
```

Interface Endpoint では、Private DNS を有効にすることで、AWS サービス名を Endpoint 用 ENI の Private IP へ名前解決できる。

```text
ssm.ap-northeast-1.amazonaws.com
↓
DNS
↓
Endpoint用ENIのPrivate IP
```

---

## 10. EC2 と Security Group の違い

誤った理解:

```text
EC2はステートレス
```

正しい理解:

```text
Security Groupはステートフル
```

意味:

```text
EC2から開始した通信
↓
相手から返ってくる応答
↓
自動的に許可される
```

そのため、EC2 用 Security Group の Inbound Rule が空でも、EC2 から開始した通信への応答は戻ってこられる。

```text
EC2
↓ HTTPS通信を開始
Interface Endpoint
↓ 応答
EC2へ戻る
```

ただし、相手側から新しく始まる通信は別扱い。

```text
相手側
↓ 新規通信
EC2
```

この場合は、EC2 用 Security Group の Inbound Rule で許可が必要。

---

## 11. Route Table の役割

S3 Gateway Endpoint では、Private Subnet 用 Route Table に S3 向けルートを追加する。

```text
Private EC2
↓
Private Subnet用Route Table
↓
S3向けPrefix List
↓
S3 Gateway Endpoint
↓
Amazon S3
```

Route Table の例:

```text
Destination: 10.0.0.0/16
Target: local

Destination: pl-xxxxxxxx
Target: vpce-xxxxxxxx
```

`pl-xxxxxxxx` は Prefix List。

```text
S3が利用するIPアドレス範囲のまとまり
```

と考えればよい。

---

## 12. SSM Agent の役割

誤った理解:

```text
SSM Agentが、
EC2へ接続してよいかお願いする
```

正しい理解:

```text
SSM Agentが、
AWS側とHTTPS通信し、
命令やセッションをやり取りする
```

流れ:

```text
AWS Systems Manager
↓
Interface Endpoint
↓
SSM Agent
↓
EC2内で処理
```

Session Manager では、SSM Agent が AWS 側と通信し、入力や出力をやり取りする。

# 最後に一文で整理

```text
Interface Endpoint
→ Subnet内にPrivate IP付きの入口を作る

Gateway Endpoint
→ Route TableにAWSサービス向け専用ルートを追加する

DNS
→ AWSサービス名をInterface EndpointのPrivate IPへ向ける

Security Group
→ 通信を許可する門であり、ステートフル

Route Table
→ 通信の宛先ごとに、次にどこへ送るか決める
```
