# EBS追加ボリューム実践：学びのまとめ

## 今回学んだこと

EC2に追加EBSをアタッチし、Ubuntu上で `/data` として使えるようにした。

さらに、`/etc/fstab` を設定することで、再起動後も自動で `/data` にマウントされることを確認した。

今回の一番大事な学びは以下。

```text
EBSをアタッチしただけでは、Linux上の保存場所としては使えない。
Linuxで使うには、ファイルシステムを作成し、ディレクトリにマウントする必要がある。
```

---

## EBSとは

EBSは、EC2に接続して使うブロックストレージサービス。

EC2を作成すると、通常はOS用のルートEBSが最初から付いている。

```text
ルートEBS
→ OSが入っているディスク
→ / として使われる

追加EBS
→ 後から追加するデータ保存用ディスク
→ 今回は /data として使った
```

---

## ルートEBSと追加EBSの違い

今回の `lsblk` では、以下のように見えた。

```text
nvme0n1
→ ルートEBS

nvme1n1
→ 追加EBS
```

ルートEBSには、UbuntuのOSやシステムファイルが入っている。

追加EBSは、後からEC2に接続したデータ保存用のディスク。

---

## アタッチとは

アタッチとは、EBSをEC2に接続すること。

```text
EBSをアタッチする
= EC2にディスクを接続する
```

アタッチすると、Ubuntu側では `/dev/nvme1n1` のようなデバイスとして見える。

ただし、この時点ではまだ普通のフォルダとしては使えない。

```text
アタッチ済み
→ Linuxはディスクの存在を認識している

未マウント
→ Linux上の保存場所としてはまだ使えない
```

---

## lsblkとdf -hの違い

今回の重要な切り分け。

```text
lsblk
→ Linuxが認識しているディスクを見る

df -h
→ マウントされ、実際に使える保存場所を見る
```

つまり、以下の状態はこう判断する。

```text
lsblkに nvme1n1 がある
df -hに /data がない
```

これは、

```text
追加EBSはアタッチされている
しかし、まだ /data にマウントされていない
```

という状態。

---

## ファイルシステムを作る意味

EBSをアタッチした直後は、Ubuntuから見ると新しい空のディスクが接続された状態。

しかし、そのままだとUbuntuはそのディスクにファイルやディレクトリをどう保存すればよいか分からない。

そこで、ファイルシステムを作成する。

```bash
sudo mkfs -t ext4 /dev/nvme1n1
```

この意味は以下。

```text
/dev/nvme1n1 というディスクを
ext4 という形式で使えるように初期化する
```

まとめると、

```text
ファイルシステムを作る
= Ubuntuがそのディスクにファイルやディレクトリを保存・管理できる形式にする
```

---

## mkfsの注意点

`mkfs` はファイルシステムを作成するコマンド。

ただし、実質的には対象ディスクを初期化する。

そのため、対象を間違えるとデータが消える。

```text
正しい対象：
/dev/nvme1n1

絶対に間違えてはいけない対象：
ルートEBS
```

今回の場合、`lsblk` でルートEBSと追加EBSを確認してから実行した。

---

## マウントとは

マウントとは、ディスクをLinux上のディレクトリに接続すること。

今回の場合は、追加EBSを `/data` にマウントした。

```bash
sudo mount /dev/nvme1n1 /data
```

この意味は以下。

```text
/dev/nvme1n1 を /data に接続する
```

これにより、`/data` 配下が追加EBS上の保存場所になる。

---

## /dataを使った理由

追加EBSのマウント先として `/data` を使った。

理由は、データ保存用ディレクトリとして分かりやすいから。

```text
/
→ OS全体のルート

/data
→ 追加データ保存用として分かりやすい
```

実務でも、追加ディスクを `/data` や `/mnt/data` のような場所にマウントすることがある。

---

## UUIDを使った理由

自動マウント設定では、デバイス名ではなくUUIDを使った。

```text
/dev/nvme1n1
→ 環境や起動順によって変わる可能性がある

UUID
→ ファイルシステムごとに付く一意のID
→ 安定して指定できる
```

そのため、`blkid` でUUIDを確認した。

```bash
sudo blkid /dev/nvme1n1
```

---

## /etc/fstabとは

`/etc/fstab` は、起動時にどのディスクをどこへマウントするかを設定するファイル。

今回、以下のような行を追加した。

```text
UUID=自分のUUID /data ext4 defaults,nofail 0 2
```

これにより、再起動後も追加EBSが自動で `/data` にマウントされる。

---

## nofailを使った理由

`nofail` は、対象のディスクが見つからない場合でも起動を止めないための設定。

```text
nofail
→ 指定したディスクがなくても起動失敗にしない
```

練習環境では、追加EBSを外したり削除したりすることがあるため、`nofail` を付けておくと安全。

---

## systemctl daemon-reloadを使った理由

`/etc/fstab` を変更した後、以下のようなメッセージが出た。

```text
mount: (hint) your fstab has been modified, but systemd still uses
       the old version; use 'systemctl daemon-reload' to reload.
```

これは、fstabを変更したが、systemdがまだ古い設定を使っているという意味。

そのため、以下を実行した。

```bash
sudo systemctl daemon-reload
```

その後、`mount -a` で再確認した。

---

## mount -aを使った理由

`mount -a` は、`/etc/fstab` に書かれている内容をもとにマウントを実行するコマンド。

```bash
sudo mount -a
```

いきなり再起動する前に、fstabの設定ミスがないか確認するために使った。

```text
mount -aでエラーが出ない
→ fstabの書き方が大きく間違っていない可能性が高い
```

---

## 再起動確認で見たこと

再起動後に、以下を確認した。

```bash
lsblk
df -h
cat /data/test.txt
```

確認した意味。

```text
lsblk
→ 追加EBSが認識されているか

df -h
→ /data にマウントされているか

cat /data/test.txt
→ 追加EBS上のデータが読めるか
```

これにより、自動マウントが成功していることを確認できた。

---

## 今回の理解まとめ

```text
EBS作成
→ AWS側でディスクを作る

アタッチ
→ EC2にディスクを接続する

lsblk
→ Linuxがディスクを認識しているか見る

mkfs
→ Ubuntuがファイルを扱える形式にする

mount
→ Linux上のディレクトリとして使えるようにする

df -h
→ 実際にマウントされているか見る

fstab
→ 再起動後も自動マウントする

reboot
→ 自動マウントの最終確認
```

---

## 一番大事なポイント

```text
アタッチされていることと、マウントされていることは別。
```

```text
lsblkに出る
→ ディスクは見えている

df -hに出る
→ 保存場所として使える
```

この違いを理解できたことが、今回の一番大きな学び。
